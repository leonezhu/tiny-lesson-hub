Lesson 12 of 12 · 产品设计

AI Agent 产品设计

当产品从"人操作工具"变成"人指挥智能体"——设计师需要掌握的新范式。

一、从 Chatbot 到 Agent：范式的转变

你可能用过 ChatGPT —— 你问，它答。这叫 Chatbot（聊天机器人）。但 AI Agent（智能体） 不一样：它能自主规划、调用工具、执行任务。

AI Agent （智能体）

一个能感知环境、做出决策、采取行动并反思结果的系统。核心能力：感知 → 规划 → 行动 → 反思 的闭环。

Chatbot 像餐厅的服务员：你点菜，他传话给厨房，然后把菜端上来。他不做决定。

Agent 像一位私人管家：你说"帮我安排下周去上海的行程"，他会自己查航班、看酒店、排会议、发提醒。他做的是一整件事，而不是一句话的回答。

生活案例

Chatbot：你说"明天天气怎样？"，它告诉你"28°C，小雨"。
Agent：你说"明天有雨，我有个户外团建"，它说"已帮你改签到室内场地 B，重新发送了邀请邮件，原场地已取消预订。"

这种转变对产品设计意味着什么？意味着你的用户不再是在"使用产品"，而是在"委托任务"。

💡 关键洞察：传统产品的交互是「人找功能」，Agent 产品的交互是「人给目标」。设计师的工作从设计按钮和流程，变成了设计目标表达和结果评估。

二、Agent 的四层架构

理解 Agent 怎么工作，才能设计好 Agent 产品。几乎所有 Agent 都包含这四个层次：

Agent 四层架构

1
感知层（Perception）
Agent 如何获取信息？通过文本输入、截图、API 数据、传感器……
设计要点：告诉用户 Agent 能看到什么、不能看到什么。
2
规划层（Planning）
把大目标拆解为小步骤。比如"写一份周报" → 收集数据 → 分析趋势 → 生成草稿 → 润色。
设计要点：让用户能看到 Agent 的计划，必要时可以修改。
3
执行层（Action）
调用工具完成任务：搜索、发邮件、操作软件、调用 API。
设计要点：每次执行前是否需要确认？高风险操作必须二次确认。
4
反思层（Reflection）
检查结果是否达标，不达标则调整策略。这是 Agent 和脚本的核心区别。
设计要点：让用户了解 Agent 是否"思考过"，增加信任感。

假设你有一个 AI Agent 产品叫 PitchBot，帮销售准备客户提案：

  • 感知：读取客户的行业、公司规模、过往合作记录（从 CRM 拉取数据）
  • 规划：分析客户痛点 → 选择成功案例 → 生成 PPT 大纲 → 起草讲稿
  • 执行：调用模板引擎生成 PPT，从素材库选配图，自动排版
  • 反思：检查提案是否覆盖了客户关心的三个 KPI，如果没有就补充

作为设计师，你要设计的是：用户如何描述客户需求？Agent 展示计划的方式？执行过程中的进度反馈？反思后的结果呈现？

三、Agent 产品的五大设计原则

Agent 产品不是简单地把 Chatbot 包装一下。它有全新的设计挑战。以下是经过实践验证的五条原则：

1. 透明性（Transparency）—— 让 Agent 的"思考过程"可见

用户最怕黑箱。当 Agent 说"我在帮你处理"，用户想知道它在做什么、为什么这么做。

反面案例

你让 Agent 整理一份竞品分析报告，它沉默了 30 秒然后给了你一个 PDF。你不知道它查了几个来源、有没有遗漏、是不是随便拼凑的。

正面设计

Agent 实时显示："正在搜索 3 个竞品官网 → 已抓取 12 页内容 → 对比功能矩阵 → 生成摘要"。用户可以随时暂停、调整方向。

2. 可控性（Controllability）—— 用户始终在驾驶座

Agent 应该像一个优秀的副驾驶，而不是自动驾驶到失控。关键设计模式：

渐进式自主（Progressive Autonomy）：新用户先让 Agent 给建议，用户确认后执行；老用户可以直接让 Agent 全权处理。

干预点设计（Intervention Points）：在关键决策节点暂停，让用户确认或修改。比如 Agent 发现数据不一致时停下来问你。

撤销能力（Undo）：Agent 做了任何操作，用户必须能一键撤回。这是信任的底线。

3. 可预测性（Predictability）—— 用户能预判 Agent 会怎么做

如果 Agent 每次的表现都不一样，用户就不会用它。一致性比聪明更重要。

⚠️ 常见陷阱：很多团队追求 Agent 的"创造力"，但产品设计师应该追求"可靠性"。一个稳定输出 80 分结果的 Agent，远比一个偶尔 100 分但经常 30 分的 Agent 更有价值。

4. 容错性（Fault Tolerance）—— 接受 Agent 会犯错

Agent 不是万能的。好的设计不是假装它不会出错，而是让错误容易发现和纠正。

A
前置确认
高风险操作前弹窗确认
B
置信度提示
"我对这个回答只有 60% 把握"

Agent 失败时自动切换备选方案

5. 关系设计（Relationship Design）—— 用户和 Agent 是合作伙伴

Agent 产品本质上是在设计一种人机关系。这种关系应该是：

  • 有边界感的亲密：Agent 了解你的偏好，但不会越界
  • 有能力的谦逊：Agent 知道自己能做什麼、不能做什麼
  • 可教育的伙伴：用户可以说"上次那样不对，这次换成……"，Agent 能学习和调整

四、Agent 产品的关键交互模式

以下是 Agent 产品中最高频的五种交互模式，每种都有具体的设计要点：

模式 1：对话式任务委托（Conversational Delegation）

用户用自然语言描述目标，Agent 理解并执行。

设计要点：

  • 提供示例指令模板，帮助用户学会"正确提问"
  • 支持多轮澄清——当用户说的不够清楚时，Agent 应该追问而非猜测
  • 显示理解确认——"你希望我帮你做 X，对吗？"
生活案例

你说："帮我安排下周的出差。"
❌ 差的 Agent：直接订了机票酒店，结果日期错了。
✅ 好的 Agent："好的！请问是去哪个城市？大概几天？有偏好的酒店吗？预算范围？"
→ 用户回答后 → "我找到了这些方案，你看看哪个合适？"

模式 2：工具编排（Tool Orchestration）

Agent 需要调用多个外部工具来完成复杂任务。

设计要点：

  • 展示 Agent 正在使用的工具列表和执行进度
  • 允许用户替换/禁用某个工具
  • 工具调用失败时，给出明确的错误原因和替代方案

模式 3：结果审查与编辑（Review & Edit）

Agent 产出结果后，用户需要审查和修改。

设计要点：

  • 提供差异对比视图（diff view），突出 Agent 做了什么改动
  • 支持逐条批准/拒绝，不必全盘接受或拒绝
  • 修改后能反向教 Agent——"下次不要这样做"

模式 4：主动提醒与建议（Proactive Assistance）

Agent 不只是被动响应，还能主动发现问题并提出建议。

生活案例

你的投资 Agent 发现某只股票连续三天异常下跌，主动推送通知："注意到 XX 股票异常波动，建议查看。原因是昨晚发布了财报，营收低于预期 15%。需要我生成详细分析报告吗？"

注意：主动提醒的关键是克制。太多打扰会让用户关闭通知，太少又失去了 Agent 的价值。

模式 5：协作式探索（Collaborative Exploration）

用户和 Agent 一起探索未知领域，Agent 提供信息、用户做判断。

设计要点：

  • 提供可视化探索界面——知识图谱、思维导图、时间线等
  • Agent 的每一次推荐都要附带信息来源
  • 支持用户在探索路径上随时转弯

五、如何衡量 Agent 产品的成功？

Agent 产品的指标体系和传统产品不同。以下是三个维度的关键指标：

维度指标说明
任务完成度Task Success RateAgent 成功完成用户目标的比例。这是最核心的指标。
Mean Tasks to Completion平均需要多少轮交互才能完成任务？越少越好。
用户信任度Acceptance Rate用户接受 Agent 建议的比例。过低说明信任不足，过高可能说明用户没仔细看。
Interruption Rate用户中途打断/纠正 Agent 的频率。
效率提升Time Saved相比手动完成，节省了多少时间？
Autonomy Level用户选择让 Agent 自主执行的比例（从低到高）。
💡 反直觉的发现：很多 Agent 产品在 Task Success Rate 很高时，用户满意度反而低。原因是用户觉得"虽然成功了，但过程太不可控"。所以过程体验和结果体验同样重要。

六、Agent 产品设计 Checklist

在设计每一个 Agent 功能之前，过一遍这份清单：

透明度：用户能否看到 Agent 正在做什么？
可控性：用户能否在过程中干预、修改、暂停？
可撤销：Agent 做的每个操作都能撤回吗？
置信度：Agent 知道自己有多确定吗？会告诉用户吗？
失败优雅：Agent 失败时，用户知道下一步该怎么办吗？
边界清晰：用户明确知道 Agent 能做什么、不能做什么吗？
可教育：用户能否告诉 Agent"上次那样不好"？
渐进自主：新手和老手是否有不同的自主级别？
一致体验：同样的输入，Agent 的输出是可预测的吗？
