数据库与SQL

TeachHub · 编程与计算机科学 · 第6课 / 共12课 · 约22分钟阅读

本课目标：理解数据库为什么是现代软件的「心脏」——它让数据持久化、可查询、可并发、可恢复。掌握关系型数据库的核心概念（表、行、主键、外键），学会用 SQL 进行增删改查（CRUD），理解事务的 ACID 特性和索引的工作原理。最终你能回答：为什么几乎所有企业应用都离不开数据库。

一、为什么需要数据库？

到目前为止，你的程序处理的数据都存在内存里——程序一关，数据就消失了。但现实世界的要求是：

  • 持久化：银行余额、订单记录必须永久保存
  • 并发：上千用户同时查询和修改，不能互相覆盖
  • 查询：从十亿条记录中秒级找到「北京、过去 7 天、金额大于 1 万」的订单
  • 安全恢复：断电、崩溃后数据不丢、不损坏

如果你用文件（如 JSON、CSV）存数据，以上每一条都会变成噩梦。数据库 (Database) 就是专门为解决这些问题而设计的数据管理系统。

数据库 ≠ 文件。数据库是一套专业软件，负责高效地存储、查询、并发控制、崩溃恢复。它让应用程序只管「我要什么数据」，不用操心「数据怎么存、怎么并发、怎么不丢」。

二、关系型数据库：用表格组织数据

最主流的数据库类型是关系型数据库（RDBMS），如 MySQL、PostgreSQL、Oracle、SQLite。它的核心思想是：用「表格」组织数据，用「关系」连接不同表。

关系型数据库 (Relational Database)
以「二维表」形式组织数据的数据库。每张表有列（字段/属性）和行（记录），表与表之间通过「外键」建立关联。1970 年由 E.F. Codd 提出，至今仍是企业数据存储的绝对主流。

来看一个电商系统的例子：

users 表

id (主键) name email
1Alicealice@mail.com
2Bobbob@mail.com

orders 表

id (主键) user_id (外键) amount date
101199.002024-01-15
102150.002024-01-20
1032200.002024-01-18

orders.user_id 是外键——它指向 users.id，建立了「订单属于用户」的关联。通过这种关联，你可以查询「Alice 的所有订单」而不需要把用户信息复制到每个订单里——这就是「关系」的力量。

主键 (Primary Key)
表中唯一标识每一行的列。每个值不能重复、不能为空。如 users.id。
外键 (Foreign Key)
表中引用另一张表主键的列，用于建立表间关联。如 orders.user_id 指向 users.id。外键保证了「引用完整性」——不能创建一个指向不存在用户的订单。

三、SQL：与数据库对话的语言

SQL (Structured Query Language) 是操作关系型数据库的标准语言。它是一种声明式语言——你描述「要什么数据」，数据库自己决定「怎么高效地找出来」（查询优化器）。核心操作分为四类，俗称 CRUD。

1. 查询 (SELECT) — 最常用的操作

-- 查询所有用户
SELECT * FROM users;

-- 查询特定列
SELECT name, email FROM users;

-- 条件查询
SELECT * FROM orders WHERE amount > 100;

-- 排序
SELECT * FROM orders ORDER BY amount DESC;

-- 聚合统计
SELECT user_id, COUNT(*) AS order_count, SUM(amount) AS total
FROM orders
GROUP BY user_id;
-- 结果：每个用户的订单数和总金额

2. 插入 (INSERT)

INSERT INTO users (name, email) VALUES ('Charlie', 'charlie@mail.com');

3. 更新 (UPDATE)

UPDATE users SET email = 'newalice@mail.com' WHERE id = 1;

4. 删除 (DELETE)

DELETE FROM orders WHERE id = 103;
SQL 的强大在于它的声明式特性。你写 SELECT name FROM users WHERE age > 18，数据库的查询优化器会自动决定用全表扫描还是索引查找——你不用关心执行细节。这正是第5课「声明式编程」思想在数据领域的体现。

四、JOIN：连接多张表

关系型数据库的精髓在于JOIN——把分散在不同表中的数据按关联条件拼在一起。

-- 查询每个订单及其对应的用户名
SELECT orders.id, users.name, orders.amount
FROM orders
JOIN users ON orders.user_id = users.id;

-- 结果：
-- 101 | Alice | 99.00
-- 102 | Alice | 50.00
-- 103 | Bob   | 200.00

JOIN 的类型：

  • INNER JOIN：只返回两表中都匹配的行（最常用）
  • LEFT JOIN：返回左表所有行，右表没有的填 NULL
  • RIGHT JOIN：返回右表所有行
  • FULL OUTER JOIN：返回两表所有行
直觉理解：INNER JOIN 像「求交集」——只有两边都有的才保留。LEFT JOIN 像「以左边为主，右边没匹配的就空着」。想象把两张表沿着外键「粘」在一起，不同的 JOIN 决定了没粘上的部分怎么处理。

五、事务与 ACID：保证数据一致性

这是数据库最强大的特性之一。假设 Alice 给 Bob 转账 100 元，操作分两步：(1) Alice 扣 100 元，(2) Bob 加 100 元。如果第一步成功、第二步失败（比如断电），Alice 的钱没了，Bob 没收到——灾难。事务 (Transaction) 就是解决这种问题的。

事务 (Transaction)
一组要么全部成功、要么全部回滚的操作单元。事务保证数据从一个一致状态转换到另一个一致状态，中间不会出现「半完成」的不一致。

事务的四个特性——ACID：

  • A — 原子性 (Atomicity)：事务是不可分割的原子单元。要么全做，要么全不做。不会出现「做了一半」。
  • C — 一致性 (Consistency)：事务执行前后，数据必须满足所有约束（主键、外键、唯一性等）。不会留下违法的数据。
  • I — 隔离性 (Isolation)：并发执行的多个事务互不干扰，仿佛是串行执行的。
  • D — 持久性 (Durability)：事务一旦提交，结果就永久保存——即使系统崩溃也不会丢失。
-- 用事务包裹转账
BEGIN TRANSACTION;
  UPDATE accounts SET balance = balance - 100 WHERE user = 'Alice';
  UPDATE accounts SET balance = balance + 100 WHERE user = 'Bob';
COMMIT;  -- 两条都成功才提交；任一失败则 ROLLBACK 回滚
ACID 是数据库区别于普通文件的根本所在。文件系统没有事务——写一半断电文件就损坏了。数据库通过预写日志 (WAL)、锁机制、MVCC 等技术，保证了即使在硬件故障下数据依然一致。

六、索引：让查询飞起来

没有索引时，SELECT * FROM users WHERE email = 'x@y.com' 要逐行扫描整张表——100 万行就比对 100 万次（O(n)）。这太慢了。

索引 (Index) 就像书的目录——它是一种额外的数据结构（通常是 B+ 树），让数据库能快速定位到目标行。

-- 在 email 列上建索引
CREATE INDEX idx_email ON users(email);

-- 现在这个查询从 O(n) 降到 O(log n)
SELECT * FROM users WHERE email = 'x@y.com';

这正是第3课「二分搜索」的工程化——B+ 树是二分搜索的多路推广，让数据库在海量数据上实现毫秒级查找。

⚠ 索引不是免费的：索引加速了读，但拖慢了写（每次插入/更新都要维护索引树）。而且索引占额外磁盘空间。所以索引要加在「经常查询但很少修改」的列上。过度索引反而降低写入性能——这是工程权衡的典型案例。

七、NoSQL：非关系型数据库的崛起

关系型数据库统治了 40 年，但互联网时代催生了 NoSQL——处理关系型数据库不擅长的场景：

  • 文档数据库（MongoDB）：存灵活结构的 JSON 文档，适合内容管理、用户画像
  • 键值数据库（Redis）：极速缓存，微秒级响应，适合会话、排行榜
  • 列族数据库（Cassandra）：海量分布式写入，适合日志、IoT 数据
  • 图数据库（Neo4j）：存关系网络，适合社交、推荐、反欺诈

NoSQL 放松了 ACID 的一致性要求（很多遵循 BASE——最终一致性），换取了更高的扩展性和灵活性。现代系统通常是混合架构：用关系型库存核心交易数据，用 Redis 做缓存，用 Elasticsearch 做全文搜索，用 MongoDB 存非结构化日志。

本课核心收获：
  1. 数据库 = 专业数据管理系统，解决持久化、并发、查询、恢复四大问题
  2. 关系型数据库用「表 + 主键 + 外键」组织数据，SQL 是操作它的声明式语言
  3. SQL 四大操作：SELECT（查）、INSERT（增）、UPDATE（改）、DELETE（删）
  4. JOIN 连接多表，事务的 ACID 保证数据一致性
  5. 索引（B+ 树）把查询从 O(n) 降到 O(log n)，但写入有代价——工程权衡

🔗 跨学科链接

  • → 图书馆学：关系型数据库的表设计源于关系代数（Codd, 1970），而关系代数与图书馆的「卡片目录」异曲同工——都是用结构化方式组织海量信息。
  • → 会计学：事务的「原子性」与会计的「复式记账」本质相同——一笔交易必须借贷平衡，不能只记一半。ACID 的概念在会计中早有雏形。
  • → 数学：SQL 的 JOIN、SELECT、WHERE 对应关系代数的笛卡尔积、投影、选择运算。SQL 本质是「关系代数的可执行版本」。
  • → 城市规划：索引类似城市的「门牌号系统」——没有门牌号（索引），快递员要逐户敲门（全表扫描）；有了门牌号，直接定位。B+ 树是「分区→街道→门牌」的层级定位推广。

🌍 现实应用

案例 1：你每次网购的背后。当你下单时，数据库用一个事务完成：扣库存、创建订单、记支付、更新用户积分——四张表要么全成功要么全回滚。如果中间断电，你的钱不会被扣但订单没生成。这就是为什么电商系统可以放心做秒杀——ACID 保证了数据绝不出错。
案例 2：银行的核心系统。全球银行每天处理数十亿笔交易，全部依赖关系型数据库的事务保证。一笔跨境汇款涉及几十个系统、上百次数据更新，但 ACID 保证最终状态一定正确。没有数据库的事务，整个金融体系无法运转。
