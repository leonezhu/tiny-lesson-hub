计算机网络：TCP/IP 到 HTTP

TeachHub · 编程与计算机科学 · 第7课 / 共12课 · 约20分钟阅读

本课目标：理解计算机网络的核心模型——数据如何从你的手机穿越半个地球到达服务器再返回。掌握 TCP/IP 分层模型、IP 地址与路由、TCP 可靠传输机制、HTTP 协议的无状态特性，建立「互联网不过是一套约定好的通信规则」的心智模型。

一、网络的核心：协议就是约定

当你打开手机上的一个网页，背后发生了一件不可思议的事——你的手机向千里之外的一台服务器发送了一条消息，服务器接收、理解、处理，然后把结果传回来，整个过程不到一秒。这一切靠的不是魔法，而是协议 (Protocol)。

协议 (Protocol)
通信双方约定好的规则集合。它规定了「数据用什么格式组织」「谁先说话」「出错怎么办」。没有协议，两台机器就像一个说中文一个说法语的人，永远无法沟通。
互联网的本质不是电缆和路由器，而是一套全球公认的通信协议——TCP/IP 协议族。硬件只是载体，协议才是灵魂。

把协议想象成「交通规则」。全世界有无数条路、无数辆车，但如果大家不约定好「靠右行驶」「红灯停绿灯行」，交通就会瞬间崩溃。网络协议就是数字世界的交通规则。

二、TCP/IP 分层模型

为了管理网络通信的复杂性，工程师们把它拆成了四层。每一层只负责一个特定任务，并把自己的结果交给下一层。这种「分而治之」的思想，就是第1课讲到的「抽象」在网络领域的体现。

四层模型一览（从上到下）

  • 应用层 (Application Layer)：你最常打交道的一层。HTTP、HTTPS、SMTP（邮件）、DNS（域名解析）都在这里。它关心的是「数据代表什么含义」。
  • 传输层 (Transport Layer)：负责端到端的数据传输。TCP 和 UDP 在这一层。它关心的是「数据是否可靠到达」。
  • 网络层 (Internet Layer)：负责寻址和路由。IP 协议在这里。它关心的是「数据该送到哪个地址」。
  • 链路层 (Link Layer)：负责物理传输。Wi-Fi、以太网、光纤在这里。它关心的是「数据怎么在相邻节点之间移动」。
类比：寄快递。应用层 = 你写的信件内容；传输层 = 快递公司保证「送货上门」；网络层 = 写在包裹上的收件地址；链路层 = 卡车和飞机的实际运输。你只管写信（应用层），不用操心卡车怎么开（链路层）。

当你发送一个 HTTP 请求时，数据会经历封装 (Encapsulation)的过程——从上往下，每一层都会在数据外面包上一层「信封」（头部信息），到了对方再从下往上一层层拆开。

# 数据封装过程（简化示意）
[应用层] GET /index.html HTTP/1.1
      ↓ 加上 HTTP 头
[传输层] TCP头 + HTTP数据
      ↓ 加上 TCP 头（端口号）
[网络层] IP头 + TCP头 + HTTP数据
      ↓ 加上 IP 头（IP地址）
[链路层] 帧头 + IP头 + TCP头 + HTTP数据
      ↓ 加上以太网帧头
分层的核心价值：每一层可以独立改进。你从 Wi-Fi 换成 5G（链路层变了），HTTP 协议（应用层）完全不需要改动。这就是分层的威力。

三、IP 地址与路由：数据怎么找到目的地？

互联网上的每台设备都有一个唯一地址——IP 地址。就像你家有一个门牌号，邮递员靠它找到你。

IPv4：32 位的地址

IPv4 地址形如 192.168.1.1，由四个 0-255 的数字组成。它本质上是一个 32 位二进制数，所以总共只有 2³² ≈ 43亿个地址——这不够全球设备用，所以有了 IPv6。

IPv6：128 位的未来

IPv6 地址形如 2001:0db8:85a3::8a2e:0370:7334，128 位地址空间，有 2¹²⁸ ≈ 3.4×10³⁸ 个地址——「足够给地球上的每一粒沙子分配一个 IP」。

路由：一站一站地转发

你的数据包不是直接「飞」到目标服务器的。它经过一系列路由器 (Router)，每一台路由器读取目标 IP 地址，决定「下一步该转给谁」，然后转发出去。这就像你问路——你不知道目的地怎么走，但每个路口的人告诉你「下一段路怎么走」。

# 路由的简化逻辑
目标 IP: 142.250.190.46（Google 服务器）

你的电脑 → 路由器A → 路由器B → 路由器C → ... → Google 服务器
  （每跳一次，路由器查表决定下一步去哪）
DNS (Domain Name System)
互联网的「电话簿」。人类记住 google.com，但机器需要 IP 地址。DNS 负责把域名翻译成 IP 地址。你输入网址时，第一步就是 DNS 查询。
DNS 查询过程：你在浏览器输入 github.com → 电脑问 DNS 服务器「github.com 的 IP 是多少？」→ DNS 服务器回答 140.82.112.3 → 电脑才向这个 IP 发起连接。整个过程通常不到 50 毫秒。

四、TCP vs UDP：可靠 vs 快速的权衡

传输层有两个核心协议，它们代表了一种经典工程权衡——可靠性与速度之间的取舍。

TCP (Transmission Control Protocol)

TCP 的核心目标是可靠传输。它保证：数据不丢、不乱、不重复、按顺序到达。代价是较慢——需要先建立连接（三次握手），需要确认每个数据包收到了，丢了要重传。

三次握手：TCP 怎么建立连接

客户端                    服务器
  |                        |
  | --- SYN（我想连接）--→  |   第1次：客户端发起
  |                        |
  | ← SYN-ACK（好的，我也准备好）--  第2次：服务器回应
  |                        |
  | --- ACK（收到，开始吧）--→ |   第3次：客户端确认
  |                        |
  | ==== 连接建立，开始传数据 ==== |

为什么要三次？因为双方都要确认「我能听到你，你也能听到我」。两次握手无法保证服务器知道客户端收到了回应。

UDP (User Datagram Protocol)

UDP 的核心目标是快速。它不管数据丢没丢、到没到，直接发出去就完事。没有连接建立，没有确认，没有重传。

什么时候用 UDP？视频通话、在线游戏、DNS 查询。这些场景中，速度比可靠性重要——丢一两帧视频无所谓，但如果为了重传等几百毫秒，画面就卡了。直播延迟就是因为 TCP 重传造成的。
TCP 像「挂号信」——保证送到，有回执；UDP 像「广播」——发出去就行，收到多少算多少。选择哪个取决于你的应用更看重「可靠」还是「实时」。

五、HTTP：Web 世界的通用语言

HTTP (HyperText Transfer Protocol) 是应用层最重要的协议。你访问的每个网页、每个 API 接口，底层几乎都是 HTTP。它定义了客户端（浏览器）和服务器之间「怎么说、怎么答」。

HTTP 的核心特征：无状态

HTTP 是无状态协议 (Stateless)——服务器不会记住你。你发了第一个请求，服务器处理完就忘了你是谁；你发第二个请求，服务器把它当作一个全新的陌生人。

但网站需要记住你的登录状态，怎么办？用 Cookie。服务器在第一次响应时附赠一个「身份牌」（Cookie），之后你每次请求都带上它，服务器靠它认出你。

HTTP 请求的结构

GET /api/users/123 HTTP/1.1      ← 方法 + 路径 + 版本
Host: example.com                 ← 头部 (Headers)
Authorization: Bearer abc123      ← 认证信息
Accept: application/json          ← 期望的返回格式

(请求体 — GET 请求通常没有)

HTTP 响应的结构

HTTP/1.1 200 OK                   ← 版本 + 状态码 + 状态文字
Content-Type: application/json    ← 头部
Content-Length: 45                ← 数据长度

{"id": 123, "name": "Alice"}      ← 响应体 (Body)

HTTP 状态码：服务器的表情包

  • 2xx 成功：200 OK（一切正常）、201 Created（创建成功）
  • 3xx 重定向：301 Moved Permanently（网址永久搬家）、302 Found（临时搬家）
  • 4xx 客户端出错：404 Not Found（找不到）、403 Forbidden（你没权限）、401 Unauthorized（没登录）
  • 5xx 服务器出错：500 Internal Server Error（服务器崩了）、503 Service Unavailable（服务器忙）
记住这个口诀：2xx 是「好的」、3xx 是「去别处」、4xx 是「你搞错了」、5xx 是「我搞砸了」。4 开头查客户端，5 开头查服务器。

HTTP 方法：CRUD 的映射

  • GET：读取数据（对应数据库的 Read）
  • POST：创建新数据（对应 Create）
  • PUT：完整替换数据（对应 Update）
  • PATCH：部分修改数据
  • DELETE：删除数据（对应 Delete）

六、HTTPS：加密的安全通道

HTTP 有一个致命问题——数据是明文传输的。你在咖啡店连了公共 Wi-Fi，同一个网络的人可以截获你的请求，看到你的密码。

HTTPS = HTTP + TLS/SSL 加密。它在 HTTP 和 TCP 之间加了一层加密，确保：

  • 机密性：别人看到的是加密后的乱码，无法解读
  • 完整性：数据在传输中被篡改会被发现
  • 身份认证：你访问的确实是真正的 github.com，而不是伪装者
HTTPS 的加密基于「公钥密码学」——服务器持有一对密钥（公钥+私钥）。公钥人人可见，用来加密；私钥只有服务器知道，用来解密。这个机制让「即使加密方法公开，数据依然安全」——这就是现代密码学的精妙之处。

七、从一个 URL 说起：完整的网络旅程

当你在浏览器输入 https://github.com/repo 并按回车，发生了什么？

  1. DNS 查询：浏览器问 DNS 服务器「github.com 的 IP 是？」→ 得到 IP
  2. TCP 连接：浏览器与该 IP 建立 TCP 连接（三次握手）
  3. TLS 握手：因为是 HTTPS，额外进行加密握手，交换密钥
  4. 发送 HTTP 请求：浏览器发送 GET /repo HTTP/1.1
  5. 服务器处理：GitHub 服务器收到请求，查数据库，生成页面
  6. 返回 HTTP 响应：服务器发送 200 OK + HTML 内容
  7. 浏览器渲染：浏览器解析 HTML，请求 CSS/JS/图片（可能又重复上述步骤）
  8. 连接关闭：数据传完后关闭 TCP 连接（四次挥手）

这一切通常在 200-500 毫秒内完成。你感受到的「网页加载」，背后是这八个步骤的高速协作。

本课核心收获：
  1. 互联网的核心是协议——一套全球公认的通信规则（TCP/IP 协议族）
  2. TCP/IP 四层模型：应用层、传输层、网络层、链路层，每层职责分明
  3. IP 负责寻址路由，DNS 把域名翻译成 IP 地址
  4. TCP 追求可靠（三次握手+重传），UDP 追求速度（发出去就不管）
  5. HTTP 是无状态协议，用 Cookie 记住用户；状态码 2xx好/3xx跳/4xx你错/5xx我错
  6. HTTPS = HTTP + 加密，保证机密性、完整性、身份认证

🔗 跨学科链接

  • → 经济学：TCP vs UDP 的取舍就是经典的「成本-效益」分析——可靠性（质量）与速度（效率）之间的权衡，经济学无处不在。
  • → 心理学：分层模型的「关注点分离」与认知心理学中的「组块化」(Chunking) 异曲同工——人脑无法一次处理 7 层细节，分层让人每次只关注一层。
  • → 语言学：协议就是「人工语言」的语法。HTTP 定义了客户端和服务器的「对话规则」，就像语法定义了句子的结构。
  • → 物理学：网络延迟的物理上限是光速——信号在光纤中以约 2/3 光速传播。纽约到东京的光纤往返至少 70ms，这是物理定律决定的，无法优化。

🌍 现实应用

案例 1：CDN（内容分发网络）。当你访问大网站时，数据不一定来自原始服务器。CDN 在全球部署缓存节点——你在中国访问的 YouTube 视频，可能来自东京节点的缓存，而不是美国总部。这就是为什么看视频不卡——距离近，延迟低。
案例 2：VPN 的原理。VPN 把你的数据加密后，「包裹」在一条新的隧道里发送。你的运营商只看到加密流量，看不到内容。VPN 服务器收到后解密，再转发到真正的目的地。本质上就是在网络层加了一个加密层。
