数据结构：数组、链表、树、图

TeachHub · 编程与计算机科学 · 第2课 / 共12课 · 约20分钟阅读

本课目标：理解数据结构为何是算法的「容器」——同样的算法，搭配合适的数据结构能让效率天差地别。掌握数组、链表、栈、队列、树、图、哈希表七大核心结构，学会用「时间/空间权衡」的思维评估它们的优劣。

一、为什么数据结构如此重要

第 1 课我们说「程序 = 算法 + 数据」。但数据不是散乱堆在一起的——数据如何组织，直接决定了算法能跑多快。这就是数据结构要解决的问题。

数据结构 = 组织和存储数据的方式。同一段数据，用不同结构存储，查找、插入、删除的效率可能相差百万倍。

经典对比：要在 10 亿个用户里查找「张三」是否存在。

  • 用无序数组：从头逐个比对，最坏要看 10 亿次 → 极慢
  • 用哈希表：直接计算「张三」的位置，一步到位 → 极快（约 1 次）
  • 用有序数组 + 二分查找：每次砍一半，约 30 次 → 很快

同样的数据，结构选对了，效率提升数千万倍。这就是为什么 Google、微信这样的系统必须精心设计数据结构。

数据结构 (Data Structure)
数据元素之间的特定关系（组织方式）的集合。它定义了数据如何存储、访问、修改，从而决定了基于它实现的算法的效率。

二、线性结构：数组与链表

最基础的数据结构是线性结构——元素一个接一个排列。其中最重要的两种是数组和链表，它们代表了两种截然不同的内存组织哲学。

数组 (Array)：连续的内存

数组把所有元素紧密排列在一块连续的内存里。每个元素紧挨着下一个，像一排连号储物柜。

# Python 列表（动态数组）
arr = [10, 20, 30, 40, 50]
print(arr[0])  # 10 —— 按索引访问，O(1)
print(arr[2])  # 30

数组的优势是随机访问极快——知道索引就能瞬间定位（O(1)）。代价是插入和删除慢：在中间插入一个元素，后面所有元素都要往后挪（O(n)）。

链表 (Linked List)：分散的节点

链表的元素（节点）分散在内存各处，每个节点除了存数据，还存一个指向下一个节点的指针，像寻宝游戏的线索：每个地点告诉你下个地点在哪。

# 链表节点的概念
class Node:
    def __init__(self, data):
        self.data = data
        self.next = None  # 指向下一个节点

# a → b → c → None
a = Node("A")
b = Node("B")
c = Node("C")
a.next = b
b.next = c

链表的优势是插入和删除快——只需改几个指针，不必挪动其他元素（O(1)，前提是已找到位置）。代价是不能随机访问——想看第 1000 个元素，必须从头顺着指针走 1000 步（O(n)）。

数组 vs 链表，是「空间换时间」的经典权衡。数组用连续空间换取快速访问；链表用额外指针空间换取灵活增删。没有「更好」，只有「更适合场景」。
现实类比：数组像一栋公寓——门牌号连续，按号找房一秒到位，但中间加塞一间房要全楼重排。链表像寻宝游戏——每个地点藏一条线索指向下个地点，插入新地点很容易（只改两条线索），但找第 100 个地点必须走完前 99 个。

三、受限的线性结构：栈与队列

栈和队列是「加了规则的线性结构」——它们限制了数据的进出方式，从而成为许多算法的核心工具。

栈 (Stack)：后进先出 LIFO

栈像一摞盘子——只能从顶部放进去、从顶部拿走。最后放进去的最先拿出来（Last In, First Out）。

stack = []
stack.append("A")  # 压入
stack.append("B")
stack.append("C")
print(stack.pop())  # C —— 最后进的先出
print(stack.pop())  # B

栈的应用：函数调用（每个函数的局部变量存在栈帧里）、撤销操作 (Undo)、括号匹配、表达式求值。当你调用一个函数，它的「执行上下文」就被压入栈；函数返回时弹出。

队列 (Queue)：先进先出 FIFO

队列像排队买票——先来的先服务（First In, First Out）。从一端进，从另一端出。

from collections import deque
q = deque()
q.append("task1")
q.append("task2")
print(q.popleft())  # task1 —— 先进先出
print(q.popleft())  # task2

队列的应用：任务调度（操作系统的进程队列）、消息队列（Kafka、RabbitMQ）、广度优先搜索 (BFS)、打印机任务。

工程案例：浏览器的「后退」按钮用栈实现——每访问一个页面就压栈，点后退就弹栈。而「打印机任务」用队列——先提交的文档先打印。理解 LIFO 和 FIFO 的区别，就理解了一大类系统设计。

四、树：层级化的组织

当数据有层级关系时，线性结构就力不从心了。这时需要树——一种层级化的数据结构，像家谱、公司组织架构、文件系统的目录。

树 (Tree)
由节点和边组成的层级结构。有一个根节点，每个节点有零个或多个子节点，没有环。树天然适合表达「一对多」的关系。

最重要的树是二叉搜索树 (Binary Search Tree, BST)——它有一个神奇的性质：左子树所有值 < 根节点 < 右子树所有值。这使得查找、插入、删除都能在 O(log n) 完成（如果树是平衡的）。

# 二叉搜索树的查找
#         8
#        / \
#       3   10
#      / \    \
#     1   6    14

def search(node, target):
    if node is None:
        return False
    if target == node.value:
        return True
    elif target < node.value:
        return search(node.left, target)   # 往左找
    else:
        return search(node.right, target)  # 往右找

# 查找 6：8→3→6，3 步找到（数组要顺序比对）

树的变种极其丰富：

  • 平衡树 (AVL, 红黑树)：自动保持平衡，保证 O(log n)。Linux 内核、C++ STL 大量使用红黑树。
  • 堆 (Heap)：用于优先队列、堆排序。Python 的 heapq。
  • 字典树 (Trie)：用于字符串前缀匹配，搜索引擎自动补全的核心。
  • B 树 / B+ 树：数据库索引的基石（MySQL、PostgreSQL）。
现实案例：你电脑的文件系统就是一棵树——根目录 / 下有 home、etc，每个目录下又有子目录和文件。这种层级结构让你能用路径（/home/user/docs/report.pdf）精确定位任何文件，无论磁盘上有多少亿个文件。

五、图：最通用的关系表达

图 (Graph) 是最通用的数据结构——它用顶点 (Vertex) 表示事物，用边 (Edge) 表示事物之间的关系。如果树是「一对多」的层级，图就是「多对多」的网络。

图 (Graph)
由顶点集和边集组成。边可以有方向（有向图）或无方向（无向图），可以带权重（如距离、成本）。图能表达任何关系网络：社交关系、道路网、网页链接。 
# 用邻接表表示图
graph = {
    "北京": ["上海", "广州"],
    "上海": ["北京", "广州", "深圳"],
    "广州": ["北京", "上海"],
    "深圳": ["上海"]
}

# 图的遍历：广度优先 (BFS) 找最短路径
from collections import deque
def bfs(graph, start):
    visited = set([start])
    queue = deque([start])
    while queue:
        node = queue.popleft()
        print(node)
        for neighbor in graph[node]:
            if neighbor not in visited:
                visited.add(neighbor)
                queue.append(neighbor)

图的应用无处不在：

  • 社交网络：微信好友关系是无向图，微博关注是有向图。
  • 地图导航：高德、Google Maps 用加权图 + Dijkstra 算法算最短路径。
  • 网页：整个互联网是有向图，PageRank 算法基于此给网页排序。
  • 推荐系统：用户-商品二部图是协同过滤的基础。
树是图的特例（无环连通图），链表是树的特例（每个节点最多一个子节点）。理解了图，你就理解了所有关系型数据的本质。

六、哈希表：用数学换速度

哈希表 (Hash Table) 是工程中最常用的数据结构之一，它用一个小技巧实现了近乎「瞬间」的查找。

核心思想：用一个哈希函数把键（如 "张三"）映射到一个数字（数组下标），然后直接把数据存在那个位置。查找时，再用同一个函数算出位置，一步到位。

# Python 字典就是哈希表
user_db = {}
user_db["张三"] = {"age": 25, "city": "北京"}
user_db["李四"] = {"age": 30, "city": "上海"}

print(user_db["张三"])  # O(1) 查找，几乎瞬间
# {'age': 25, 'city': '北京'}

哈希表的平均查找、插入、删除都是 O(1)——这是数据结构里的「圣杯」。但代价是：

  • 哈希冲突：两个不同的键算出同一个位置。需要用链地址法或开放寻址解决。
  • 无序：不能按顺序遍历（除非用 LinkedHashMap）。
  • 最坏情况：如果哈希函数很差，所有键都冲突，退化成 O(n)。
工程现实：Python 的 dict、Java 的 HashMap、Redis、Memcached、区块链的状态存储——底层都是哈希表。互联网基础设施几乎有一半建立在哈希表之上，因为它把「用名字找东西」这件事做到了极致速度。

七、如何选择数据结构：权衡的艺术

没有「最好」的数据结构，只有「最适合当前场景」的。选择的依据是你最常做的操作：

  • 频繁按索引访问？→ 数组
  • 频繁头尾增删？→ 链表 / 双端队列
  • 需要后进先出？→ 栈（撤销、递归）
  • 需要先进先出？→ 队列（任务调度）
  • 需要按键瞬间查找？→ 哈希表
  • 需要保持有序 + 快速查找？→ 平衡二叉搜索树
  • 需要表达层级关系？→ 树
  • 需要表达任意关系网络？→ 图
数据结构的选择本质是权衡：时间 vs 空间，读取 vs 写入，有序 vs 无序。成熟的工程师不追求「最优」，而是追求「在当前约束下最合适」。
本课核心收获：
  1. 数据结构决定了算法的效率——同样的数据，结构选对，效率可能差百万倍
  2. 线性结构（数组/链表）是基础，栈和队列是加了规则的线性结构
  3. 树表达层级关系，图表达任意网络关系——图是最通用的结构
  4. 哈希表用数学换速度，平均 O(1)，是工程界的「圣杯」
  5. 选择数据结构的本质是权衡：根据最常做的操作来选

🔗 跨学科链接

  • → 数学：图论是离散数学的核心分支，欧拉 1736 年解决「哥尼斯堡七桥问题」开创了图论。树和图的性质都用数学严格证明。
  • → 经济学：「权衡」思维与经济学的「机会成本」完全一致——选了数组的高效访问，就放弃了链表的灵活增删，没有免费的午餐。
  • → 心理学：人类大脑天然用「树状分类」组织知识（动物→哺乳动物→猫科）。数据结构的「树」正是对人类认知结构的工程化模拟。
  • → AI：决策树、随机森林、图神经网络 (GNN) 都是建立在树和图结构之上的机器学习模型。理解数据结构是理解现代 AI 的前提。

🌍 现实应用

案例 1：Git 的底层是树和图。Git 的每个 commit 是一个节点，commit 之间通过 parent 指针连成有向无环图 (DAG)。分支只是指向某个 commit 的指针。整个版本历史就是一棵不断生长的树——这就是为什么 Git 能高效地管理分支、合并、回溯。理解了 DAG，你就理解了 Git 的灵魂。
案例 2：社交网络的好友推荐。微信「可能认识的人」功能，底层是图算法：你的好友和 X 的共同好友越多（图上你们和 X 之间路径越短、越多），系统越可能推荐 X。这种「图上的相似度」计算，正是 LinkedIn、Facebook 推荐系统的核心。
