算法思维：排序、搜索、复杂度

TeachHub · 编程与计算机科学 · 第3课 / 共12课 · 约22分钟阅读

本课目标：建立「算法思维」——学会用步骤、效率和代价的眼光看待问题。掌握排序与搜索两大经典算法族，理解大O记号如何量化算法的效率差异，并能根据数据规模选择合适的算法。最终你能回答：为什么有些程序慢到无法使用，而有些程序在百万级数据上依然飞快。

一、什么是算法思维？

第1课我们说过：算法是「解决问题的明确步骤」。但仅有步骤还不够——同一个问题，不同的解法效率可以相差亿万倍。算法思维，就是在解决问题时不仅想「能不能做」，更要想「做得够不够快、够不够省」。

算法思维 = 在解决问题时，同时关注正确性（结果对不对）和效率（资源花得多不多）。

举个例子：你要在字典里查一个词。一本十万词的字典，你怎么查？

  • 方法 A：从第一页翻到最后一页，逐个比对。——一定能找到，但平均要翻五万页。
  • 方法 B：翻到字典中间，看当前词比目标大还是小，然后在对应的一半里继续折半。——最多翻 17 次就能找到。

这就是算法思维的威力：同样的输入，方法 B 比方法 A 快了约三千倍。在十万条数据上，一个是「秒级」，一个是「全天候跑不完」。

时间复杂度 (Time Complexity)
衡量算法运行时间随输入规模增长的趋势。它不关心具体的毫秒数，只关心「数据变大时，时间怎么变」。

二、大O记号：衡量效率的通用语言

要比较算法，我们需要一种与硬件、语言无关的度量。这就是大O记号 (Big-O Notation)。它描述的是：当输入规模 n 趋向无穷大时，算法所需操作次数的增长阶。

大O记号回答的是：「数据量翻倍、翻十倍、翻一万倍时，这个算法会慢多少倍？」它关注的是趋势，不是常数。

常见的时间复杂度，从快到慢：

复杂度 名称 n=10 n=1000 n=100万
O(1)常数111
O(log n)对数31020
O(n)线性101000100万
O(n log n)线性对数33100002000万
O(n²)平方100100万1万亿
O(2ⁿ)指数1024∞∞

请仔细看最后一列：n=100万时，O(n²) 已经是一万亿次操作（按每秒一亿次算要 3 小时），而 O(2ⁿ) 直接爆炸到宇宙都算不完。这就是为什么选择算法比优化代码细节重要得多——一个 O(n²) 的程序再怎么微优化，也追不上一个 O(n log n) 的程序。

直觉记忆：O(1) 是「开门拿东西」（不管房子多大），O(log n) 是「翻字典」（每次砍一半），O(n) 是「逐个点名」，O(n²) 是「两两握手」（n 个人每人握 n 次），O(2ⁿ) 是「所有子集」（每多一个元素，方案数翻倍）。

三、排序：计算机科学的「Hello World」

排序是算法教学中最经典的课题，因为它足够简单、又有丰富的效率差异，完美展示了「同一问题不同解法的天壤之别」。

1. 冒泡排序 (Bubble Sort) — O(n²)

最直观但最慢的排序。思路：反复遍历列表，每次比较相邻两个元素，如果顺序错了就交换。像气泡一样，大的元素会逐渐「浮」到末尾。

def bubble_sort(arr):
    n = len(arr)
    for i in range(n):
        for j in range(0, n - i - 1):
            if arr[j] > arr[j + 1]:
                arr[j], arr[j + 1] = arr[j + 1], arr[j]
    return arr

print(bubble_sort([5, 2, 9, 1, 5, 6]))
# [1, 2, 5, 5, 6, 9]

两层嵌套循环，所以是 O(n²)。排序 10000 个元素要做一亿次比较——这就是它慢的原因。

2. 归并排序 (Merge Sort) — O(n log n)

分治法的典型代表。思路：把数组对半切开，递归地排序两半，然后合并两个已排序的半段。

def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])
    right = merge_sort(arr[mid:])
    return merge(left, right)

def merge(a, b):
    result = []
    i = j = 0
    while i < len(a) and j < len(b):
        if a[i] <= b[j]:
            result.append(a[i]); i += 1
        else:
            result.append(b[j]); j += 1
    result.extend(a[i:])
    result.extend(b[j:])
    return result

为什么是 O(n log n)？每层合并需要 O(n)，而递归深度是 log n 层（每次砍半），相乘得 O(n log n)。排序 100 万个元素约 2000 万次操作——比冒泡快了五万倍。

分治法 (Divide and Conquer)：把大问题拆成小问题，分别解决，再合并结果。这是计算机科学最重要的思想之一——归并排序、快速排序、二分搜索、FFT、MapReduce 都源自这个思想。

3. 快速排序 (Quick Sort) — 平均 O(n log n)

实际工程中最常用的排序。思路：选一个「基准」元素，把比它小的放左边、大的放右边，然后对左右两段递归排序。

def quick_sort(arr):
    if len(arr) <= 1:
        return arr
    pivot = arr[len(arr) // 2]
    left = [x for x in arr if x < pivot]
    middle = [x for x in arr if x == pivot]
    right = [x for x in arr if x > pivot]
    return quick_sort(left) + middle + quick_sort(right)

快排在大多数实际数据上比归并更快（常数因子小），但最坏情况会退化到 O(n²)（比如每次选到最大或最小的基准）。Python 的 sorted() 底层用的是 Timsort——一种融合了归并和插入排序的混合算法，专门针对真实世界数据优化。

工程案例：Google 曾经面临一个挑战——每天要对几十亿个网页做排序来计算排名。如果用 O(n²) 算法，光排序就要跑几年。换成 O(n log n) 后，同样的任务几分钟搞定。算法的选择，直接决定了一个产品「能不能上线」。

四、搜索：找东西的艺术

1. 线性搜索 (Linear Search) — O(n)

最简单的搜索：从头到尾逐个比对。

def linear_search(arr, target):
    for i, v in enumerate(arr):
        if v == target:
            return i
    return -1

简单直接，但对无序列表必须 O(n)。n=100 万时要平均比对 50 万次。

2. 二分搜索 (Binary Search) — O(log n)

前提：列表必须已排序。每次取中间元素，根据比较结果砍掉一半，直到找到或确认不存在。

def binary_search(arr, target):
    lo, hi = 0, len(arr) - 1
    while lo <= hi:
        mid = (lo + hi) // 2
        if arr[mid] == target:
            return mid
        elif arr[mid] < target:
            lo = mid + 1
        else:
            hi = mid - 1
    return -1

# 只能用于已排序数组
print(binary_search([1, 3, 5, 7, 9, 11], 7))  # 3
二分搜索是 log n 的奇迹：100 万元素的列表，最多只要 20 次比较。每多一倍数据，只多一次操作。这是「指数级减半」的力量。

二分搜索的精髓是「每次淘汰一半」。这个思想远不止于搜索——求平方根、找方程的根、优化机器学习的超参数、数据库索引查找，都基于同样的「折半」策略。

生活类比：猜数字游戏——我想一个 1 到 100 的数，你猜我告诉你是大了还是小了。如果你从 1 开始逐个猜，平均要 50 次。如果你每次猜中间值（50、然后 25 或 75……），最多 7 次就能猜中。这就是二分搜索。

五、空间复杂度：内存也要花钱

算法消耗的资源不止是时间，还有内存。空间复杂度衡量算法运行时占用的额外内存随 n 的增长趋势，同样用大O记号。

  • 冒泡排序：原地交换，额外空间 O(1)
  • 归并排序：需要 O(n) 额外空间存放合并结果
  • 递归调用：每层递归占用一份栈空间，深度 d 的递归占 O(d)
时间和空间经常是一对权衡 (Time-Space Tradeoff)。用更多内存可以换来更快的速度——例如哈希表用 O(n) 空间把查找降到 O(1)。

六、最好、最坏、平均：同一个算法的三副面孔

大O通常描述最坏情况（保证算法在任何输入下都不会更差）。但实际中我们也关心平均情况。以快速排序为例：

  • 最好：每次基准恰好把数组平分 → O(n log n)
  • 平均：基准大致均匀分布 → O(n log n)
  • 最坏：每次基准都是最大/最小 → O(n²)

工程上常通过「随机选基准」来避免最坏情况，让快排在概率意义上稳定在 O(n log n)。这就是为什么 Python、Java 的标准库都用改进版快排。

工程案例：Netflix 的推荐系统每天要为上亿用户排序候选视频。如果用 O(n²) 算法，单次推荐就要几分钟。改用 O(n log n) 后降到毫秒级——这才是「即时推荐」的物理基础。算法不是象牙塔里的数学，它直接决定用户体验。

七、如何选择算法？

没有「最好」的算法，只有「最适合当前场景」的算法。选择时要考虑：

  • 数据规模：n 小时简单算法（如插入排序）反而更快，因为常数因子小
  • 数据特征：几乎已排序的数据用插入排序接近 O(n)；随机数据用快排
  • 内存限制：嵌入式设备可能无法承受 O(n) 额外空间的归并排序
  • 稳定性需求：归并排序是稳定排序（相等元素保持原顺序），快排不是
  • 是否需要实时响应：最坏情况敏感的场景要避免可能退化到 O(n²) 的算法
本课核心收获：
  1. 算法思维 = 同时关注正确性和效率。同一问题不同解法效率可差亿万倍
  2. 大O记号衡量「增长趋势」，与硬件语言无关。O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ)
  3. 排序：冒泡 O(n²)、归并 O(n log n)、快排平均 O(n log n)。真实系统多用 Timsort
  4. 搜索：无序用线性 O(n)，有序用二分 O(log n)。二分是「折半淘汰」的普适思想
  5. 时间与空间常需权衡。没有万能算法，只有场景适配

🔗 跨学科链接

  • → 数学：大O记号源自数学分析，是描述函数渐近行为的严格工具。log 函数的「增长极慢」特性，是信息论（熵、编码长度）和计算机科学共同的基石。
  • → 经济学：「时间-空间权衡」与经济学中的「资源替代」如出一辙——用一种资源（内存）替代另一种资源（时间）来优化总成本。
  • → 生物学：归并排序的「分治」思想，与细胞分裂、神经网络层级处理、基因组的分块组装异曲同工——大自然也偏爱分治。
  • → 管理/运筹学：排序算法是「调度问题」的特例。快递分拣、工厂流水线、医院急诊分诊，本质都是某种排序问题。

🌍 现实应用

案例 1：数据库索引。当你在一个百万行的用户表里查「id=12345」的用户，数据库不是逐行扫描（O(n)），而是用 B+ 树索引——一种二分搜索的推广。这让查找从「几秒」降到「微秒」。整个现代数据库的性能，本质上建立在本课的算法之上。
案例 2：Git 的 diff 算法。当你 git diff 时，Git 用的是 Myers 算法——一种基于动态规划的最长公共子序列算法，复杂度 O(ND)（N、D 是文本长度和差异数）。这就是为什么 Git 能秒级显示几千行代码的差异。
