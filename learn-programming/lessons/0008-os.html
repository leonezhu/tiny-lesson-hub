操作系统原理

TeachHub · 编程与计算机科学 · 第8课 / 共12课 · 约22分钟阅读

本课目标：理解操作系统——那个你每天都在用却很少思考的「大管家」。掌握进程与线程的本质区别、CPU 调度原理、内存管理与虚拟内存、文件系统抽象，以及并发带来的经典问题（死锁、竞态条件）。建立「操作系统是一个资源管理者」的心智模型。

一、操作系统：看不见的大管家

你可能从来没「见过」操作系统——你看不到它的界面，却无时无刻不在依赖它。当你打开浏览器、听音乐、写代码，这些程序之所以能同时运行而不互相干扰，全靠操作系统在幕后做调度。

操作系统 (Operating System, OS)
一组管理计算机硬件资源、为应用程序提供运行环境的系统软件。它是硬件和应用之间的「中间人」——应用不需要知道硬盘怎么转、屏幕怎么亮，只需向 OS 请求服务。

操作系统的核心职责可以用一个词概括：资源管理。一台计算机的 CPU、内存、硬盘、网卡都是有限的，但有几十甚至几百个程序想同时使用它们。操作系统的工作就是公平、高效地分配这些资源。

操作系统管理四大核心资源：CPU（计算）、内存（存储）、I/O 设备（输入输出）、文件（数据持久化）。每一个都对应一个子系统。

二、进程与线程：谁在执行代码？

进程 (Process)：程序的运行实例

当你双击一个程序图标，操作系统会创建一个进程。进程是程序的一次执行——它拥有独立的内存空间、打开的文件、网络连接等资源。

关键区分：程序和进程不同。程序是磁盘上的一个文件（静态的），进程是这个文件被加载到内存后正在运行的实例（动态的）。一个程序可以同时运行多个进程——比如你开了 3 个 Chrome 窗口，就是 3 个进程。

线程 (Thread)：进程内的执行单元

一个进程内部可以有多个线程。线程是 CPU 调度的最小单位。同一个进程内的线程共享内存空间，但各自有独立的执行栈。

# 进程与线程的关系

进程 A（Chrome）
├── 线程 1：渲染页面
├── 线程 2：播放音频
├── 线程 3：处理网络请求
└── 线程 4：垃圾回收
    （共享进程 A 的内存空间）

进程 B（Word）
├── 线程 1：编辑界面
└── 线程 2：自动保存
    （独立于进程 A 的内存空间）

进程 vs 线程：经典对比

  • 内存隔离：进程之间内存隔离（A 不能直接访问 B 的内存），线程共享内存（同进程内线程可以直接读写彼此的数据）
  • 创建开销：创建进程开销大（要复制内存空间），创建线程开销小（只需分配栈）
  • 通信方式：进程通信需要 IPC（管道、消息队列等），线程通信直接读写共享变量
  • 崩溃影响：一个进程崩溃不影响其他进程，但一个线程崩溃可能导致整个进程崩溃
经验法则：需要强隔离、高安全 → 用进程；需要高并发、低开销 → 用线程。现代 Web 服务器（如 Nginx）用「多进程+多线程」混合模型。

三、CPU 调度：一台 CPU 怎么「同时」跑一百个程序？

你的电脑可能只有 4-8 个 CPU 核心，但你同时开着浏览器、音乐播放器、代码编辑器、微信——它们看起来都在运行。这是怎么做到的？答案是时分复用 (Time Sharing)。

CPU 给每个进程分配一小段时间（称为时间片，通常几毫秒），用完后切换到下一个进程。由于切换极快（每秒上千次），人类感觉不到中断，以为所有程序「同时」运行。

# 时分复用的简化示意（时间 →）

进程A ████│进程B ████│进程C ██│进程A ████│进程B ██│...
            ↑ 切换    ↑ 切换  ↑ 切换

实际上 CPU 任何时刻只运行一个进程
但切换速度极快 → 人眼感觉是「并行」的
并发 (Concurrency) vs 并行 (Parallelism)
并发：多个任务在交替执行（同一时刻只有一个在运行，但切换极快）。并行：多个任务真正同时运行（需要多个 CPU 核心）。单核 CPU 只能并发，多核 CPU 才能真正并行。

调度算法：谁先跑？谁等？

当有 100 个进程等着用 CPU，操作系统需要决定运行顺序。常见的调度策略有：

  • 先来先服务 (FCFS)：谁先到谁先跑。简单但可能让短任务等很久。
  • 短任务优先 (SJF)：预计时间最短的先跑。平均等待时间最优，但需要预知任务长度。
  • 轮转调度 (Round Robin)：每人跑一个时间片，轮流来。公平，是分时系统的核心策略。
  • 优先级调度：高优先级先跑。可能导致低优先级任务「饿死」（永远轮不到）。
现实类比：医院急诊室。FCFS = 按挂号顺序看病；优先级调度 = 重症患者优先（但有被「插队」的风险）；轮转 = 医生每个病人看 5 分钟轮流来。实际系统通常是多种策略的混合。

四、内存管理与虚拟内存

物理内存的困境

一台电脑有 16GB 物理内存，但如果你开了浏览器（占 4GB）、IDE（占 2GB）、游戏（占 8GB）……加起来超过 16GB 了。为什么系统没崩溃？因为虚拟内存。

虚拟内存的魔法

操作系统给每个进程一种错觉——让它以为自己独占了整个内存空间（比如 64 位系统上进程以为自己有 2⁶⁴ 字节）。实际上，物理内存远不够分配。OS 通过页表 (Page Table) 在虚拟地址和物理地址之间做映射。

虚拟内存的核心思想：每个进程拥有独立的、连续的虚拟地址空间。OS 负责把虚拟地址映射到物理地址。进程不需要关心物理内存的真实布局——这就是「抽象」的完美体现。

分页 (Paging)

内存被分成固定大小的块（通常 4KB），称为页 (Page)。虚拟内存中的「虚拟页」和物理内存中的「物理页帧」之间通过页表映射。当进程访问的虚拟页不在物理内存中时，触发缺页中断 (Page Fault)，OS 从硬盘把数据加载进来。

换页 (Swapping)

当物理内存不够时，OS 会把暂时不用的页写到硬盘上的交换区 (Swap)，腾出空间给急需的进程。需要时再从 Swap 读回来。由于硬盘速度远慢于内存，频繁换页会导致系统极慢——这就是「卡到死」的根源。

# 虚拟内存映射（简化）

进程 A 的虚拟地址空间          物理内存
┌─────────────┐              ┌─────────────┐
│ 虚拟页 0 ───────→ 映射到 ──→│ 物理帧 2    │
│ 虚拟页 1 ───────→ 映射到 ──→│ 物理帧 5    │
│ 虚拟页 2 ───────→ 在 Swap → │ (硬盘上)    │
│ 虚拟页 3 ───────→ 映射到 ──→│ 物理帧 8    │
└─────────────┘              └─────────────┘

五、文件系统：硬盘的抽象

硬盘本质上是一堆磁性扇区——物理层面只有「第 12345 号扇区」这种编号。但操作系统给你提供了文件 (File) 和目录 (Directory) 的抽象，让你看到的是 /home/user/photo.jpg 这样友好的路径。

文件系统 (File System)
操作系统管理存储设备上数据的方式。它把物理扇区组织成文件和目录的层次结构，提供创建、读取、写入、删除等操作。常见的有 ext4（Linux）、NTFS（Windows）、APFS（macOS）。

文件系统的核心任务之一是空间分配——一个文件可能跨多个不连续的扇区。常见策略有：

  • 连续分配：文件占一段连续的扇区。读取快，但容易产生碎片。
  • 链式分配：每个扇区指向下一个，像链表。灵活但随机访问慢。
  • 索引分配：用一个索引表记录文件的所有扇区位置。ext4/NTFS 使用的方案。
为什么删除文件后还能恢复？因为删除通常只是把文件目录项标记为「可用」，数据本身还在扇区里——直到被新数据覆盖。这就是为什么「彻底删除」需要多次覆盖写入。

六、并发问题：当多个线程同时操作共享数据

并发编程是计算机科学中最难的领域之一。当多个线程同时读写同一块内存时，如果不加以控制，就会产生竞态条件 (Race Condition)。

竞态条件

# 经典的竞态条件：两个线程同时给 balance 加 100

balance = 0

线程 A: 读取 balance (0)        ← 读
线程 B: 读取 balance (0)        ← 读
线程 A: 计算 0 + 100 = 100      ← 算
线程 B: 计算 0 + 100 = 100      ← 算
线程 A: 写入 balance = 100      ← 写
线程 B: 写入 balance = 100      ← 写

# 结果：balance = 100（应该是 200！）
# 这就是竞态条件——结果取决于线程执行的交错顺序

互斥锁 (Mutex)

解决方案是互斥锁——确保同一时间只有一个线程能进入关键区域。线程 A 操作 balance 时先「锁住」，线程 B 只能等 A 解锁后才能操作。

balance = 0
lock = Mutex()

# 线程 A 和 B 都执行：
lock.acquire()           # 获取锁（其他人等）
balance = balance + 100  # 关键区域
lock.release()           # 释放锁

# 结果：balance = 200 ✅

死锁 (Deadlock)

但锁带来了新的灾难——死锁。两个线程各持有一把锁，同时等待对方释放另一把锁，永远卡住。

# 死锁的经典场景

线程 A：持有锁1，等待锁2
线程 B：持有锁2，等待锁1

→ 两个线程永远等待 → 死锁 💀
死锁的四个必要条件（ Coffman 条件）：互斥、持有并等待、不可抢占、循环等待。打破任何一个就能避免死锁。实践中最常用的策略是「固定顺序获取锁」——所有线程都按相同顺序请求锁，就不会循环等待。

七、系统调用：应用程序与 OS 的接口

应用程序不能直接操作硬件。当你想读文件、发网络请求、分配内存，你必须通过系统调用 (System Call) 向操作系统「请求服务」。操作系统是唯一能直接控制硬件的软件。

# 常见系统调用

read(fd, buffer, size)    # 读文件
write(fd, buffer, size)   # 写文件
open(path, flags)         # 打开文件
fork()                    # 创建新进程
exec(program)             # 执行新程序
malloc(size)              # 分配内存
socket()                  # 创建网络连接
为什么不能让程序直接操作硬件？安全与隔离。如果任何程序都能直接读写硬盘，一个恶意程序可以删掉所有数据。操作系统通过系统调用实现了权限控制——程序只能做 OS 允许它做的事。这就是为什么你的浏览器不会直接删除你的文件。
本课核心收获：
  1. 操作系统是资源管理者，管理 CPU、内存、I/O、文件四大核心资源
  2. 进程 = 运行中的程序（独立内存），线程 = 进程内执行单元（共享内存）
  3. CPU 通过时分复用让多任务「同时」运行——并发 ≠ 并行
  4. 虚拟内存让每个进程以为自己独占整个内存，靠页表映射到物理内存
  5. 文件系统把物理扇区抽象为文件和目录的层次结构
  6. 并发编程要警惕竞态条件和死锁——锁是工具，也是风险来源

🔗 跨学科链接

  • → 经济学：CPU 调度算法与经济学的「资源分配」高度相似——公平与效率的权衡、优先级与饥饿问题，都是经典经济学命题。
  • → 政治学：进程隔离 vs 线程共享，对应「联邦制」vs「单一制」——前者强调独立自治（安全），后者强调资源共享（效率）。
  • → 物理学：虚拟内存的「换页」与热力学的「熵」——系统越复杂，越需要消耗能量（I/O 开销）来维持秩序。
  • → 心理学：时分复用与人类的「注意力切换」异曲同工——你以为在「多任务」，实际上是在快速切换注意力，只是你自己意识不到。

🌍 现实应用

案例 1：Docker 和容器。Docker 容器本质是利用了 Linux 的 namespace（命名空间隔离）和 cgroup（资源限制）特性。它让进程以为自己运行在独立系统中，但底层共享同一个操作系统内核。比虚拟机轻量得多——这就是为什么容器化成为现代云原生应用的标准。
案例 2：手机杀后台应用真的省电吗？不一定。操作系统有自己的内存管理策略——空闲内存会被用来做缓存（加速下次启动）。你手动杀掉的应用，下次打开需要重新加载（消耗 CPU 和电池）。除非内存严重不足，否则让 OS 自己管理更好。
