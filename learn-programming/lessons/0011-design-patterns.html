设计模式：复用前人的智慧

TeachHub · 编程与计算机科学 · 第11课 / 共12课 · 约22分钟阅读

本课目标：理解什么是设计模式，为什么它们是软件工程的「词汇表」。掌握四大经典模式——单例、观察者、策略、工厂——的意图、结构和适用场景。学会区分模式的两面性：何时该用，何时不该用，避免「过度设计」的陷阱。

一、什么是设计模式？

假设你是一名建筑设计师。你发现，无论设计什么样的房子，总有一些问题反复出现：门朝哪开？楼梯放哪？客厅和卧室怎么分隔？经过几十年的实践，行业沉淀出了一套标准方案——比如「开放式厨房适合小户型」「卧室远离入口保证安静」。这些方案不是法律规定，而是被验证过的经验模板。

软件设计模式 (Design Pattern) 就是编程领域的「标准方案」。

设计模式 (Design Pattern)
在特定场景下，对反复出现的设计问题的一种可复用的解决方案。它不是一段可以直接复制的代码，而是一种设计思路的模板——告诉你「遇到这类问题时，通常应该这样组织代码」。

这个概念由四位作者——Erich Gamma、Richard Helm、Ralph Johnson、John Vlissides（合称「四人帮」GoF）——在 1994 年的经典著作《设计模式》中系统化。书中归纳了 23 种面向对象的设计模式，从此成为软件工程师的共同语言。

设计模式最大的价值不是「教你写代码」，而是提供一套共同的词汇。当你说「这里用观察者模式」，对方立刻理解你要表达的设计意图——不需要你解释几十行代码。模式让设计沟通从「描述实现」升级为「描述意图」。
为什么需要模式？想象你每次盖房子都从零开始琢磨「墙怎么搭、屋顶怎么防水」。前辈们已经踩过无数坑、总结出最优解，你为什么不站在他们的肩膀上？设计模式就是前辈工程师留给我们的「踩坑笔记」。

二、模式的三大分类

GoF 把 23 种模式分为三类，按粒度从大到小排列：

  • 创建型模式 (Creational)：关注「如何创建对象」。把对象的创建过程抽象出来，让系统不依赖于具体的类。代表：单例、工厂、建造者。
  • 结构型模式 (Structural)：关注「如何把类或对象组合成更大的结构」。代表：适配器、装饰器、代理。
  • 行为型模式 (Behavioral)：关注「对象之间如何分工和通信」。代表：观察者、策略、状态、命令。
这三个分类回答了软件设计的三个核心问题：怎么造（创建）、怎么搭（结构）、怎么协作（行为）。记住这个框架，你就能快速判断某个模式属于哪类、解决什么层次的问题。

本课我们深入讲解四个最常用、最有代表性的模式——每种类型各取一个或两个——让你建立对模式的「手感」。

三、单例模式 (Singleton)

解决的问题

有些东西，全局只应该有一个实例。比如：数据库连接池、日志记录器、系统配置对象。如果这些对象被创建了多个副本，会导致资源浪费、状态不一致、甚至程序崩溃。

单例模式 (Singleton)
保证一个类只有一个实例，并提供一个全局访问点。属于创建型模式。

结构

核心思路：把构造函数设为私有（外部无法 new），在类内部持有一个静态实例，通过一个静态方法返回它。

class Logger:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.logs = []
        return cls._instance

    def log(self, msg):
        self.logs.append(msg)
        print(f"[LOG] {msg}")

# 无论创建多少次，都是同一个对象
a = Logger()
b = Logger()
print(a is b)  # True —— 同一个实例
现实类比：一个国家只有一个「总统」职位。无论谁当总统，对外都通过「总统办公室」这个唯一接口沟通。你不能自己 new 一个总统出来——必须通过宪法规定的流程（静态方法）获取当前的那一位。

注意事项

单例是最容易被滥用的模式。许多初学者觉得「全局只要一个，用单例多方便」，结果把所有东西都变成单例，代码变成了披着面向对象外衣的全局变量——难以测试、难以维护。

单例的本质是全局状态。全局状态是测试和并发的噩梦——单例持有的状态在测试之间无法隔离，在多线程下需要加锁。能用依赖注入解决的，就别用单例。

四、观察者模式 (Observer)

解决的问题

当一个对象的状态变化需要通知一堆其他对象时，怎么做？

最笨的办法是：被通知的对象轮流去「问」——「你变了吗？你变了吗？」。这叫轮询，效率极低。更好的办法是：状态一变，主动「喊一声」，关心的人自己来听。

观察者模式 (Observer)
定义对象间一对多的依赖关系：当一个对象（被观察者 / Subject）的状态发生变化时，所有依赖于它的对象（观察者 / Observer）都会自动收到通知并更新。属于行为型模式。

结构

# 被观察者：气象站
class WeatherStation:
    def __init__(self):
        self._observers = []
        self._temperature = 0

    def subscribe(self, observer):
        self._observers.append(observer)

    def unsubscribe(self, observer):
        self._observers.remove(observer)

    def set_temperature(self, temp):
        self._temperature = temp
        self._notify_all()

    def _notify_all(self):
        for obs in self._observers:
            obs.update(self._temperature)

# 观察者：显示屏
class PhoneDisplay:
    def update(self, temperature):
        print(f"📱 手机显示：当前温度 {temperature}°C")

class WindowDisplay:
    def update(self, temperature):
        print(f"🪟 窗口显示：当前温度 {temperature}°C")

station = WeatherStation()
station.subscribe(PhoneDisplay())
station.subscribe(WindowDisplay())
station.set_temperature(25)
# 📱 手机显示：当前温度 25°C
# 🪟 窗口显示：当前温度 25°C
现实类比：微信公众号。你关注（subscribe）了一个公众号，它发文章时你会自动收到推送——你不需要每隔五分钟去刷新看它有没有更新。公众号是被观察者，你是观察者。这是典型的「发布-订阅」机制。

为什么重要

观察者模式是现代软件中出现频率最高的模式之一。几乎所有涉及「事件」的系统都用它：

  • GUI 框架：按钮点击事件通知所有监听器
  • 前端框架：Vue/React 的响应式数据变化触发视图更新
  • 消息队列：Kafka、RabbitMQ 的发布订阅模型
  • Node.js 的 EventEmitter、Python 的 signal、C# 的 event
观察者模式的核心价值是解耦：被观察者不需要知道有谁在观察自己、观察者有几个、它们是谁。它只负责「状态变了，喊一声」。这种松耦合让系统可以灵活扩展——加一个观察者，不需要修改被观察者的代码。

五、策略模式 (Strategy)

解决的问题

假设你在开发一个电商系统，结账时需要根据用户类型计算折扣：普通用户打 9 折、VIP 打 8 折、超级会员打 7 折。最直觉的写法是一堆 if-else：

# ❌ 糟糕的写法：策略和业务逻辑混在一起
def calculate_price(user_type, price):
    if user_type == "normal":
        return price * 0.9
    elif user_type == "vip":
        return price * 0.8
    elif user_type == "super":
        return price * 0.7
    # 以后每加一种用户类型，就要改这里

问题在于：每加一种折扣规则，就要修改这个函数。if-else 会越来越长，函数承担了太多职责，违反了「开闭原则」（对扩展开放，对修改关闭）。

策略模式 (Strategy)
定义一系列算法，把它们各自封装成独立的类（策略），使它们可以互相替换。策略模式让算法的变化独立于使用算法的客户端。属于行为型模式。

结构

# 抽象策略：定义接口
from abc import ABC, abstractmethod

class DiscountStrategy(ABC):
    @abstractmethod
    def calculate(self, price):
        pass

# 具体策略：各自封装一种算法
class NormalDiscount(DiscountStrategy):
    def calculate(self, price):
        return price * 0.9

class VIPDiscount(DiscountStrategy):
    def calculate(self, price):
        return price * 0.8

class SuperDiscount(DiscountStrategy):
    def calculate(self, price):
        return price * 0.7

# 上下文：使用策略
class Checkout:
    def __init__(self, strategy):
        self._strategy = strategy

    def set_strategy(self, strategy):
        self._strategy = strategy

    def final_price(self, price):
        return self._strategy.calculate(price)

# 使用
checkout = Checkout(NormalDiscount())
print(checkout.final_price(100))  # 90.0

checkout.set_strategy(VIPDiscount())
print(checkout.final_price(100))  # 80.0
现实类比：导航软件的路线规划。从 A 到 B 可以「驾车」「步行」「骑行」「公交」——每种方式是一个策略。你只需要选择策略，导航用该策略算路线。加一种「飞行」策略，不需要改原有代码——符合开闭原则。
策略模式的本质是用多态替代条件分支。与其写一堆 if-else 判断走哪条路，不如把每条路封装成对象，运行时动态切换。这是面向对象设计消除 if-else 地狱的经典武器。

六、工厂模式 (Factory)

解决的问题

假设你的程序需要创建多种「形状」对象：圆形、方形、三角形。如果到处直接 Circle()、Square()，一旦需求变化（比如改成需要传参数、或换一个实现类），所有调用处都要改。

工厂模式把「创建对象」这件事集中到一个地方，让调用者不关心对象怎么造，只关心拿到对象后能用。

工厂模式 (Factory)
定义一个用于创建对象的接口，让子类决定实例化哪一个类。工厂方法使一个类的实例化延迟到其子类。属于创建型模式。

结构

from abc import ABC, abstractmethod

# 产品接口
class Shape(ABC):
    @abstractmethod
    def draw(self):
        pass

# 具体产品
class Circle(Shape):
    def draw(self):
        print("● 画一个圆")

class Square(Shape):
    def draw(self):
        print("■ 画一个方形")

class Triangle(Shape):
    def draw(self):
        print("▲ 画一个三角")

# 工厂：集中管理创建逻辑
class ShapeFactory:
    @staticmethod
    def create(shape_type):
        if shape_type == "circle":
            return Circle()
        elif shape_type == "square":
            return Square()
        elif shape_type == "triangle":
            return Triangle()
        raise ValueError(f"未知形状: {shape_type}")

# 调用者不需要知道具体类名
s1 = ShapeFactory.create("circle")
s2 = ShapeFactory.create("square")
s1.draw()  # ● 画一个圆
s2.draw()  # ■ 画一个方形
现实类比：快递公司就是一个「包裹工厂」。你只需要说「寄到北京」，快递公司内部决定用哪家承运商、什么车型——你不需要关心这些细节。如果某天承运商换了，你的寄件方式不变。

工厂 vs 直接 new

初学者常问：「为什么不直接 Circle()？多此一举？」答案在于变化的隔离。当创建逻辑变得复杂——需要读配置、需要缓存、需要根据运行时条件选择不同实现——工厂就把这些复杂度封装起来，调用方代码完全不用动。

工厂模式的核心价值：封装变化。它把「创建对象」和「使用对象」分离，让系统的某一部分变化时，不影响其他部分。这是依赖倒置原则的体现——高层模块不应该依赖低层模块，二者都应该依赖抽象。

七、其他常见模式速览

除了上面四个，还有许多模式在实践中频繁出现。这里做一个速览，帮助你建立全景认知：

装饰器模式 (Decorator)

意图：动态地给一个对象添加职责，而不改变其接口。比继承更灵活。

例子：给一杯咖啡动态加「牛奶」「糖」「奶泡」——每加一层就是一个装饰器。Python 的 @decorator 语法就是这个名字的来源。

适配器模式 (Adapter)

意图：把一个类的接口转换成客户端期望的另一个接口。让原本不兼容的类能一起工作。

例子：你的笔记本只有 USB-C 接口，但鼠标是 USB-A 的——需要一个「转接头」。适配器模式就是这个转接头。

代理模式 (Proxy)

意图：为其他对象提供一个代理，以控制对这个对象的访问。

例子：远程代理（RPC 客户端）、缓存代理、权限校验代理。你调用的是代理，代理帮你转发或拦截。

命令模式 (Command)

意图：把「请求」封装成对象，从而可以用不同的请求对客户进行参数化、排队请求、记录日志、支持撤销。

例子：文本编辑器的「撤销 / 重做」功能。每个操作都是一个命令对象，存在历史栈里。

八、模式的另一面：过度设计

学完模式，最容易犯的错误是拿着锤子找钉子——看到什么都想套个模式。这是初学者的通病，被称为「模式病」。

模式不是越多越好。每一个模式都引入了额外的类、接口、间接层。简单的问题用简单的方案解决——只有一个 if-else 的时候，不需要策略模式；只有两三种对象的时候，不需要工厂。

判断何时该用模式，有一个朴素的原则：等痛点出现再重构。当你发现某段代码反复修改、难以扩展、耦合严重——这时再引入模式来解决具体的问题。而不是一开始就为「将来可能的需求」堆砌模式。

工程经验：很多资深工程师的忠告是——先用最简单的方式实现，遇到第三次重复时再抽象（Rule of Three）。过早抽象比不抽象更危险，因为你预测的未来需求往往是错的。

九、设计原则：模式背后的道

模式是「术」，模式背后还有「道」——那就是面向对象设计的六大原则（SOLID + 两个补充）。理解了原则，你甚至能自己「发明」模式：

  • 单一职责原则 (SRP)：一个类只应该有一个引起它变化的原因。
  • 开闭原则 (OCP)：对扩展开放，对修改关闭。加功能时尽量加新代码，不改老代码。
  • 里氏替换原则 (LSP)：子类必须能替换父类而不影响程序正确性。
  • 接口隔离原则 (ISP)：客户端不应该依赖它不需要的接口。
  • 依赖倒置原则 (DIP)：依赖抽象，不依赖具体实现。
每一个设计模式，本质上都是这五个原则在特定场景下的应用。策略模式体现了开闭原则和依赖倒置；工厂模式体现了依赖倒置；观察者模式体现了开闭原则。先理解原则，模式就是水到渠成的事。
本课核心收获：
  1. 设计模式是反复出现的设计问题的可复用方案，更是工程师之间的「共同词汇」
  2. 三大分类：创建型（怎么造）、结构型（怎么搭）、行为型（怎么协作）
  3. 单例保证全局唯一实例，但要警惕全局状态的测试和并发陷阱
  4. 观察者实现一对多通知，是事件驱动、响应式系统的基石
  5. 策略用多态替代条件分支，工厂封装对象的创建过程
  6. 模式是把双刃剑：解决真实痛点是利器，过度设计是负担

  • → 建筑学：设计模式的概念直接来自建筑学家 Christopher Alexander 的《建筑模式语言》——他提出「模式」是反复出现的空间问题的通用解法。软件领域借用了这个思想。

  • → 经济学：「过度设计」本质是「过早投资」，类似经济学的「期权价值」——保持灵活性（简单方案）比提前锁定（复杂架构）更值钱，当未来不确定性高时。

🌍 现实应用

案例 1：前端框架的响应式系统。Vue 的数据变化能自动触发视图更新，底层就是观察者模式。每个数据属性是一个被观察者，依赖它的组件是观察者。数据变了，组件自动重新渲染——你不需要手动调用刷新。
案例 2：Python 的 logging 模块。Python 标准库的 logging.getLogger() 就是单例模式——无论你在哪个文件调用，拿到的都是同一个 logger 实例，保证所有日志走统一的配置和输出通道。
