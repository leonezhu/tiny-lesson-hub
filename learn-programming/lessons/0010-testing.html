测试驱动开发

TeachHub · 编程与计算机科学 · 第10课 / 共12课 · 约20分钟阅读

本课目标：理解软件测试的本质——它不是「找 Bug」，而是「证明代码做了正确的事」。掌握单元测试、集成测试、TDD 的红-绿-重构循环，以及 Mock 的概念。建立「没有测试的代码是遗留代码」的心智模型。

一、为什么需要测试？

很多初学者认为：「我写的代码肯定是对的，为什么要写测试？」这是一个危险的思维。回顾第1课的核心原理——计算机是确定性机器——程序出错一定是逻辑错了，不是机器错了。测试就是你验证「逻辑是否正确」的系统性方法。

软件测试 (Software Testing)
通过运行程序、检查其行为是否符合预期，来发现缺陷的过程。测试不是「证明程序没有 Bug」（这在数学上不可能），而是「证明程序在某些情况下确实做了正确的事」。
Dijkstra 的名言：「测试只能证明 Bug 的存在，不能证明 Bug 不存在。」测试让你对「已覆盖的场景」有信心，但不能保证所有场景都没问题。

二、测试金字塔：三层结构

测试不是一刀切的。业界有一个经典模型——测试金字塔 (Test Pyramid)，将测试分为三个层次：

1. 单元测试 (Unit Test)——塔基，最多

测试最小的代码单元（一个函数、一个方法）。它应该是孤立的——不依赖数据库、网络、文件系统。运行极快（毫秒级），数量最多。

# 单元测试示例（Python pytest）

def add(a, b):
    return a + b

def test_add():
    assert add(2, 3) == 5       # 正数相加
    assert add(-1, 1) == 0      # 正负相加
    assert add(0, 0) == 0       # 零
    assert add(0.1, 0.2) == 0.3 # 浮点数

2. 集成测试 (Integration Test)——塔身，中等

测试多个模块协作是否正确。比如「数据库 + 业务逻辑 + API 返回」一起跑。它比单元测试慢，因为涉及真实的外部依赖。

# 集成测试示例：测试 API + 数据库

def test_create_user_api(client, db):
    response = client.post('/api/users', json={
        'name': 'Alice',
        'email': 'alice@example.com'
    })
    assert response.status_code == 201
    # 验证数据真的写入了数据库
    user = db.query(User).filter_by(email='alice@example.com').first()
    assert user is not None
    assert user.name == 'Alice'

3. 端到端测试 (End-to-End Test)——塔尖，最少

测试整个系统从头到尾——模拟真实用户的操作流程（打开浏览器→点击按钮→验证页面）。它最接近真实场景，但也最慢、最脆弱。

金字塔的比例：典型项目应该有 70% 单元测试、20% 集成测试、10% 端到端测试。塔基要宽（快且多），塔尖要窄（慢且少）。倒金字塔（大量 E2E，少量单元测试）是反模式——慢且脆弱。

三、TDD：测试驱动开发

测试驱动开发 (Test-Driven Development, TDD) 是一种开发方法论，由 Kent Beck 提出。它的核心是：先写测试，再写代码。这看似反直觉，但有着深刻的工程价值。

红-绿-重构 (Red-Green-Refactor)

TDD 的核心循环只有三步：

# TDD 循环

🔴 RED（红）
  → 先写一个测试，描述你想要的功能
  → 运行测试 → 必然失败（因为功能还没实现）
  → 这验证了「测试本身是有效的」

🟢 GREEN（绿）
  → 写最少量的代码让测试通过
  → 不要过度设计，只让它通过
  → 运行测试 → 通过 ✅

🔵 REFACTOR（重构）
  → 在测试保护下，改善代码质量
  → 每次重构后都运行测试，确保没有破坏
  → 测试就是你的安全网
TDD 的哲学：如果你不能为一个功能写测试，说明你还没想清楚这个功能到底该做什么。写测试的过程本身就是设计的过程——它强迫你思考接口的输入和输出。

TDD 完整示例

需求：写一个判断回文字符串的函数 is_palindrome(s)。

# 步骤1：🔴 RED — 先写测试

def test_is_palindrome():
    assert is_palindrome("racecar") == True
    assert is_palindrome("hello") == False
    assert is_palindrome("") == True       # 空字符串
    assert is_palindrome("a") == True      # 单字符

# 运行测试 → 失败！is_palindrome 还不存在

# 步骤2：🟢 GREEN — 最简实现

def is_palindrome(s):
    return s == s[::-1]

# 运行测试 → 全部通过 ✅

# 步骤3：🔵 REFACTOR — 改进（如需要）
# 比如：忽略大小写、去除空格
def is_palindrome(s):
    cleaned = s.lower().replace(" ", "")
    return cleaned == cleaned[::-1]

# 补充测试，然后运行 → 通过 ✅

四、测试的核心原则：AAA 模式

几乎所有测试都遵循 Arrange-Act-Assert (AAA) 模式：

  • Arrange（准备）：设置测试所需的数据和状态
  • Act（执行）：调用被测试的函数
  • Assert（断言）：检查结果是否符合预期
# AAA 模式示例

def test_discount():
    # Arrange
    price = 100
    discount_rate = 0.2

    # Act
    result = calculate_discount(price, discount_rate)

    # Assert
    assert result == 80

五、边界值与边界测试

Bug 最容易出现在边界——空值、零、最大值、最小值、临界条件。好的测试会专门覆盖这些边界。

# 边界值测试清单

def test_find_max_boundaries():
    # 正常情况
    assert find_max([1, 3, 2]) == 3
    # 边界：空列表
    assert find_max([]) is None
    # 边界：只有一个元素
    assert find_max([42]) == 42
    # 边界：全部相同
    assert find_max([5, 5, 5]) == 5
    # 边界：负数
    assert find_max([-3, -1, -2]) == -1
经验法则：永远测试空值、零值、单元素、重复元素、负数。80% 的 Bug 出现在这些边界上，而不是正常路径。

六、Mock：隔离依赖的利器

单元测试的核心原则是隔离——只测试目标函数，不测试它的依赖。但如果你的函数调用了数据库、网络 API、文件系统怎么办？用 Mock（模拟对象）。

Mock（模拟对象）
一个「假的」对象，它假装是真实依赖（如数据库），但行为完全可控。你预先设定「当被调用时返回什么」，让测试不依赖真实的外部系统。 
# Mock 示例（Python unittest.mock）

from unittest.mock import Mock

# 被测试的函数：从 API 获取用户信息后计算年龄
def get_user_age(user_id, api_client):
    user = api_client.fetch_user(user_id)  # 依赖外部 API
    return user['age']

# 测试时，不真正调用 API，而是 Mock 它
def test_get_user_age():
    # 创建 Mock 对象
    mock_api = Mock()
    mock_api.fetch_user.return_value = {'name': 'Alice', 'age': 30}

    # 用 Mock 代替真实 API
    age = get_user_age(123, mock_api)

    assert age == 30  # ✅ 不依赖网络，瞬间完成
Mock 的价值：让测试快（不需要网络/数据库）、稳定（不受外部服务波动影响）、可复现（同样的输入永远给同样的结果）。这三点与第1课「计算机是确定性机器」完美呼应——Mock 让测试环境变为确定性的。

七、测试覆盖率：有用但不要迷信

测试覆盖率 (Coverage) 衡量「有多少代码行被测试执行过」。100% 覆盖率意味着每一行代码至少被一个测试跑过。

高覆盖率 ≠ 高质量。你可以写出 100% 覆盖率但没有断言的测试——每行代码都被执行了，但没人检查结果对不对。覆盖率告诉你「哪里没测」，但不能告诉你「测得好不好」。

合理的覆盖率目标：

  • 核心业务逻辑：90%+
  • 工具函数/库：95%+
  • UI 层：50-70%（UI 测试脆弱且维护成本高）
  • 配置/脚手架代码：不需要测试

八、测试的好处：超越「找 Bug」

写测试的最大价值，其实不是「发现 Bug」，而是：

  • 重构的安全网：有测试，你才敢放心改代码。没测试的代码，没人敢动。
  • 活文档：测试就是代码的使用说明书——读测试就能理解函数该怎么用。
  • 设计反馈：如果某个函数很难测试，通常是设计有问题（依赖太多、职责不清）。测试是设计的「第一用户」。
  • 回归防护：修好一个 Bug 后写测试，防止它未来再次出现。
本课核心收获：
  1. 测试不是证明「没有 Bug」，而是证明「某些场景下做了正确的事」
  2. 测试金字塔：70% 单元测试 + 20% 集成测试 + 10% 端到端测试
  3. TDD = 红（写测试，失败）→ 绿（写代码，通过）→ 重构（改善质量）
  4. 测试遵循 AAA 模式：Arrange（准备）→ Act（执行）→ Assert（断言）
  5. Mock 让单元测试隔离外部依赖，保证快速、稳定、可复现
  6. 测试的最大价值是重构安全网和设计反馈，不只是找 Bug

🔗 跨学科链接

  • → 科学方法：测试与科学实验高度相似——假设（期望结果）、实验（运行代码）、观察（断言）。TDD 的红-绿循环就是「假设-验证」循环。
  • → 心理学：「确认偏差」让人倾向于只测自己觉得对的场景。好的测试要刻意寻找「可能出错的地方」——这是反直觉的批判性思维。
  • → 法学：测试覆盖率和断言强度，就像法律中的「证据充分性」。有目击者（被执行）不等于定罪（断言通过）——证据链必须完整。
  • → 经济学：测试是「保险」——付出当前成本（写测试的时间），换取未来风险的降低（回归 Bug 的损失）。和所有保险一样，关键是成本与收益的平衡。

🌍 现实应用

案例 1：CI/CD 流水线中的自动化测试。每次你提交代码（第9课学的 Git），CI 系统会自动运行全部测试。如果任何测试失败，代码就不允许合并到主线。这就是为什么大公司能保证每天部署数百次——测试自动化是持续交付的基础。
案例 2：「遗留代码」的定义。Michael Feathers 在《修改代码的艺术》中给出定义：「没有测试的代码就是遗留代码。」——不管它多新。因为没测试的代码你不敢改，不敢改的代码就会逐渐腐化。测试是代码「活着」的证明。
