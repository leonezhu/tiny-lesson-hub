函数式编程

TeachHub · 编程与计算机科学 · 第5课 / 共12课 · 约20分钟阅读

本课目标：理解函数式编程（FP）的核心理念——把计算视为数学函数的求值，而非状态的修改。掌握纯函数、不可变性、高阶函数三大支柱，理解 map/filter/reduce 如何替代循环，以及为什么 FP 在并发、大数据、AI 时代重新崛起。最终你能回答：为什么「函数无副作用」如此重要。

一、命令式 vs 声明式：两种思维范式

你已经学过 OOP——它把数据和行为打包成对象。函数式编程则走上了一条截然不同的路：它把一切计算都看作函数的求值，就像数学里的 f(x) = x² + 1。

先看一个对比例子。任务：把列表中所有偶数平方后求和。

命令式写法（告诉计算机「怎么做」每一步）：

numbers = [1, 2, 3, 4, 5, 6]
result = 0
for n in numbers:
    if n % 2 == 0:
        result += n ** 2
print(result)  # 4 + 16 + 36 = 56

函数式写法（描述「要什么」，而非怎么做）：

numbers = [1, 2, 3, 4, 5, 6]
result = sum(map(lambda x: x ** 2, filter(lambda x: x % 2 == 0, numbers)))
print(result)  # 56

# 或用列表推导式（Python 的函数式糖）
result = sum(x ** 2 for x in numbers if x % 2 == 0)
命令式编程关注「怎么做」（循环、赋值、状态变化）；声明式/函数式编程关注「要什么」（用函数描述转换关系）。函数式代码更接近数学定义，更易于推理和并行。

二、纯函数：FP 的基石

纯函数 (Pure Function)
满足两个条件的函数：(1) 给定相同的输入，永远返回相同的输出；(2) 不产生任何副作用——不修改外部状态、不读写文件、不打印、不改变参数。 
# 纯函数：只依赖输入，不碰外部
def add(a, b):
    return a + b

# 不纯的函数：依赖外部变量（每次调用结果可能不同）
total = 0
def add_to_total(x):
    total += x       # 修改了外部状态！
    return total

# 不纯的函数：有副作用
def greet(name):
    print(f"Hello, {name}")  # 副作用：输出到屏幕
    return name

纯函数的价值是什么？可预测、可测试、可并行。

  • 可预测：相同的输入永远得到相同输出——没有「隐藏依赖」，代码行为一目了然
  • 可测试：不需要 mock 外部状态，直接传输入断言输出
  • 可并行：多个纯函数可以放心并发执行，因为它们不会互相干扰
  • 可缓存：同样的输入算过一次，结果可以永久缓存（记忆化 memoization）
纯函数是数学意义上的「函数」——它只是从输入到输出的映射，不改变世界的任何其他东西。这种「无副作用」让代码变得极其容易推理。

三、不可变性：数据一旦创建就不变

在 FP 中，数据一旦创建就不会被修改。需要「改变」时，你不是改原数据，而是创建一份新的副本。

# 可变（命令式思维）
scores = [90, 85, 78]
scores.append(92)       # 原地修改
scores[0] = 95          # 原地修改
# scores 现在是 [95, 85, 78, 92]

# 不可变（函数式思维）
scores = (90, 85, 78)   # 元组，不可变
new_scores = scores + (92,)    # 创建新元组，原元组不变
# scores 仍是 (90, 85, 78)
# new_scores 是 (90, 85, 78, 92)

为什么要不可变？因为可变状态是 bug 的温床。当一个对象在多处被引用时，任何一处修改都会影响所有持有引用的代码——这种「远距离副作用」是大型系统中最难追踪的 bug 来源。

工程案例：React 的核心设计理念就是「状态不可变」。你永远不直接修改 state，而是调用 setState 创建新状态。React 拿新旧状态做对比（浅比较），就能高效判断需要重新渲染哪些组件。如果允许直接修改，React 就无法检测变化——这就是不可变性的实战价值。

四、高阶函数：函数是一等公民

在 FP 中，函数是「一等公民」——它可以像数据一样被赋值、传递、返回。高阶函数就是接收函数作为参数、或返回函数的函数。

高阶函数 (Higher-Order Function)
接收一个或多个函数作为参数，或者返回一个函数的函数。map、filter、reduce 是最经典的高阶函数。

map：对每个元素施加变换

numbers = [1, 2, 3, 4, 5]
squares = list(map(lambda x: x ** 2, numbers))
# [1, 4, 9, 16, 25]

# 等价的列表推导式
squares = [x ** 2 for x in numbers]

filter：按条件筛选

numbers = [1, 2, 3, 4, 5, 6]
evens = list(filter(lambda x: x % 2 == 0, numbers))
# [2, 4, 6]

# 等价的列表推导式
evens = [x for x in numbers if x % 2 == 0]

reduce：把列表归约为单个值

from functools import reduce

numbers = [1, 2, 3, 4, 5]
product = reduce(lambda a, b: a * b, numbers)
# 1*2=2 → 2*3=6 → 6*4=24 → 24*5=120
print(product)  # 120
map / filter / reduce 是 FP 的「三剑客」。它们让你不再写显式的循环，而是用「数据流水线」描述转换——这是声明式编程的核心。现代语言的流式 API（Java Stream、JS Array 方法、Rust 迭代器）都源于此。

五、函数组合：像乐高一样拼装

当每个函数都是纯函数且只做一件事时，你可以把它们组合成更复杂的逻辑——就像用管道把简单的零件接成流水线。

def double(x): return x * 2
def add_one(x): return x + 1
def square(x): return x ** 2

# 组合：先加1，再平方，再翻倍
def compose(f, g):
    return lambda x: f(g(x))

transform = compose(double, compose(square, add_one))
print(transform(3))  # (3+1)²×2 = 16×2 = 32

这就是 Unix 哲学的核心：cat file | grep "error" | sort | uniq -c——每个命令是一个纯函数，用管道（|）组合起来。函数组合让复杂逻辑由简单零件搭建，每个零件可独立测试和复用。

六、递归：FP 的循环替代品

纯函数式语言（如 Haskell）甚至没有循环语句——它们用递归替代。递归是一个函数调用自己，把问题分解成更小的同类问题。

# 用递归计算阶乘
def factorial(n):
    if n <= 1:
        return 1           # 基准情况（终止条件）
    return n * factorial(n - 1)  # 递归调用

print(factorial(5))  # 5×4×3×2×1 = 120

# 用递归遍历树结构
def count_tree(node):
    if node is None:
        return 0
    return 1 + count_tree(node.left) + count_tree(node.right)
递归的精髓：把大问题化成「同样的更小问题 + 一步组合」。只要有明确的基准情况（终止条件），递归就能优雅地解决树、图、分治等问题。

但要注意：递归过深会导致栈溢出（Python 默认递归深度约 1000）。很多语言提供「尾递归优化」来避免这个问题，Python 没有内置支持，所以实际工程中 Python 会用循环替代深层递归。

七、FP 在现代工程中的崛起

FP 并非新事物——它的理论基础（Lambda 演算）诞生于 1930 年代，比计算机本身还早。但它近年来强势回归，原因是：

  • 多核并发：纯函数无副作用，天然适合并行——不需要锁，不会数据竞争
  • 大数据：Hadoop 的 MapReduce、Spark 的 RDD 都基于 map/reduce 模型
  • 前端革命：React/Redux 的「不可变状态 + 纯函数 reducer」彻底改变了前端架构
  • AI/ML：TensorFlow、PyTorch 的计算图本质是函数式的数据流
工程案例：WhatsApp 曾用 Erlang（纯函数式语言）支撑了每秒 200 万条消息、仅 50 名工程师维护。Erlang 的 actor 模型 + 不可变消息 + 纯函数，让它能在海量并发下几乎不出现数据竞争 bug——这是命令式语言极难达到的。这就是为什么电信、金融交易系统偏爱 FP。

八、OOP 与 FP：互补而非对立

OOP 和 FP 不是非此即彼。现代最佳实践是混合使用：

  • 用 OOP 建模系统的结构（对象、组件、模块边界）
  • 用 FP 编写对象内部的逻辑（纯函数、不可变转换）

Scala、Rust、现代 JavaScript、TypeScript、Kotlin 都融合了两种范式。Python 也从 3.0 开始大量引入函数式特性。未来的趋势不是「哪个范式更好」，而是「在合适的场景用合适的工具」。

本课核心收获：
  1. FP 把计算视为数学函数求值，关注「要什么」而非「怎么做」
  2. 纯函数：相同输入永远相同输出，无副作用——可预测、可测试、可并行
  3. 不可变性：数据创建后不变，需要修改就创建副本——消除远距离副作用
  4. 高阶函数（map/filter/reduce）替代显式循环，描述数据转换流水线
  5. FP 在并发、大数据、前端领域强势回归，与 OOP 互补而非对立

🔗 跨学科链接

  • → 数学：FP 直接源于 Lambda 演算（Church, 1936）——一种用函数定义计算的形式系统。它与图灵机等价，是理论计算机科学的两大基石之一。
  • → 物理学：「不可变性」类似热力学中的状态函数——系统的状态是确定的、可重现的。纯函数不改变世界，就像可逆过程不增加熵。
  • → 语言学：函数组合类似句法树——简单的词（小函数）通过语法规则（组合）构成复杂的句子（大函数）。FP 的表达力来自组合规则，正如语言的表达力来自语法。
  • → 烹饪：map/filter/reduce 就像流水线厨房——洗菜（filter 去掉坏的）、切菜（map 变换形状）、炖煮（reduce 合并成菜）。每一步都是纯转换，不回头修改前面的步骤。

🌍 现实应用

案例 1：Excel 的数组公式。Excel 的 SUM(IF(A1:A10>0, A1:A10, 0)) 本质就是 filter + reduce。整个电子表格可以看作一个巨大的函数式数据流——每个单元格是一个纯函数，输入是其他单元格，输出是计算结果。改一个输入，所有依赖它的单元格自动重算。这就是 FP 的威力被包装成亿万用户都能用的工具。
案例 2：Netflix 的推荐流水线。Netflix 用 Spark 的函数式 API 处理海量用户观看数据：events.filter(是否观看).map(提取特征).reduceByKey(聚合).collect()。每一步都是纯转换，可以分布到上千台机器并行执行——不需要担心数据竞争，因为数据不可变。这就是大数据时代 FP 成为主流的原因。
