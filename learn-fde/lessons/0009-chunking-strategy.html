第 9 课 — Chunking Strategy

RAG 全链路的第一步：如何切分文档，决定了召回率的上限

一、Chunking 是什么——为什么它如此重要

RAG（检索增强生成）系统的第一步是 Chunking：将大文档拆分成小片段（chunks），以便后续做 embedding 和存入向量数据库。

很多人以为 chunking 只是"把长文本切碎"，但事实远比这复杂——

核心原则：chunk 要足够大，以包含有意义的信息；又要足够小，以保证检索性能和低延迟。这个平衡点没有标准答案，取决于数据性质和查询复杂度。

Chunking 的质量直接决定了 RAG 系统的召回率上限。一个糟糕的 chunking 策略会让后续所有环节（embedding、检索、reranking）都事倍功半。

❓ 测验 1：如果一个 chunk 切得太小（比如只有 50 tokens），最可能出现什么问题？

  • ✗ 不对。chunk 越小，数量越多，存储和检索反而可能更快。
  • ✓ 正确！太小的 chunk 无法承载完整语义，embedding 向量无法准确表达原文含义。
  • ✗ 不相关。chunk 大小主要影响召回阶段，不直接影响 LLM 推理速度。

二、Fixed-size Chunking：最佳起点

Pinecone 在 2025 年的最新指南中明确指出：fixed-size chunking 在大多数场景下是最好的起点，应该先从这里开始，只在证明不足时才迭代到更复杂的策略。

关键参数：

参数推荐值说明
Chunk size128 ~ 1024 tokens细粒度语义用 128/256；保留上下文用 512/1024
Overlap10% ~ 20%防止边界处信息丢失——这是最容易忽略的细节
Token 估算按 embedding model 实际 tokenizer不同模型的 token 化方式不同，不能简单按字符计数

为什么 overlap 必不可少？想象一个关键词恰好出现在两个 chunk 的边界上——如果没有重叠，这个关键词在一次嵌入时会被截断，导致检索失败。

❓ 测验 2：设置 chunk overlap 的主要目的是什么？

  • ✗ 重叠部分确实是重复的，不是新增信息。
  • ✓ 正确！重叠确保跨边界的语义片段在至少一个 chunk 中被完整包含。
  • ✗ 恰恰相反，overlap 会增加总 token 数和计算量。

三、Semantic Chunking：让边界跟随语义

Fold-size chunking 的缺陷在于：它不管内容是什么，一刀切。而 semantic chunking 通过计算相邻句段之间的语义距离来动态确定 chunk 边界。

工作流程：

  1. 将文档拆分为句子
  2. 将句子分组（每组 N 个相邻句子）
  3. 为每组生成 embedding
  4. 计算相邻组 embedding 之间的余弦距离
  5. 当距离超过阈值时，划分 chunk 边界

优势：每个 chunk 内的句子讨论同一主题，语义一致性高，特别适合知识密集型文档（技术手册、法律文件等）。

代价：需要额外的 embedding 调用成本，且阈值选择需要调参。

❓ 测验 3：以下哪种文档最适合使用 semantic chunking 而非 fixed-size？

  • ✗ 短文本语义结构简单，fixed-size 完全够用。
  • ✓ 正确！法律合同包含大量独立条款，语义切换频繁，semantic chunking 能更好地保持每条款完整性。
  • ✗ 独立注释之间无依赖关系，不需要语义分析。

四、Contextual Retrieval：Anthropic 的革命性方案

Anthropic 在 2024 年 9 月提出了 Contextual Retrieval，这是目前最前沿的 chunking 增强方法。核心创新非常简单却极其有效：

original_chunk = "The company's revenue grew by 3% over the previous quarter." contextualized_chunk = "This chunk is from an SEC filing on ACME corp's Q2 2023 performance; \ previous quarter's revenue was $314M. The company's revenue grew by 3% over the previous quarter."

做法：在 embedding 之前，用 LLM 为每个 chunk 生成一段 50-100 token 的上下文描述，前置到 chunk 前面。这样即使 chunk 本身缺少上下文，embedding 也能捕捉到全局语义。

实验结果（跨多个领域）：

  • 仅 Contextual Embeddings：检索失败率从 5.7% → 3.7%（降低 35%）
  • + Contextual BM25：失败率降至 2.9%（降低 49%）
  • + Reranking：失败率降至 1.9%（降低 67%）

而且，利用 Claude 的 prompt caching，百万文档 token 的上下文生成成本仅约 $1.02。

❓ 测验 4：Contextual Retrieval 的核心思想是什么？

  • ✗ 不是增大 chunk，而是在 chunk 前加一段 LLM 生成的上下文描述。
  • ✓ 正确！这是 Contextual Retrieval 的核心创新——让 embedding 模型看到全局语境。
  • ✗ 不涉及多次 embedding，是一次 embedding + 一次 LLM 生成。

五、Hybrid Search：BM25 + Semantic 是生产标配

传统 RAG 只用 embedding 做语义搜索，但 Anthropic 的实验证明 BM25 在精确术语匹配上远优于 embedding。

典型场景：用户查询 "Error code TS-999"——embedding 模型可能返回关于 error codes 的一般性内容，但 BM25 能精确匹配 "TS-999" 这个字符串。

生产级方案应同时维护两种索引：

  • Embedding index：捕捉语义相似性
  • BM25 index：基于 TF-IDF 的精确实体/术语匹配
  • Rank fusion：合并两种结果并去重

❓ 测验 5：为什么 Hybrid Search 需要同时使用 embedding 和 BM25？

  • ✗ 速度不是主要原因，两者速度差异不大。
  • ✓ 正确！语义搜索找"意思相近的"，BM25 找"字面匹配的"，合在一起覆盖更全面。
  • ✗ BM25 和 reranking 是两个不同的阶段，BM25 不能替代 reranking。

六、Chunk Expansion：检索时的后处理技巧

即使选择了最优的 chunking 策略，检索到的 top-K chunks 可能仍缺少足够的上下文供 LLM 理解。

Chunk expansion 策略：在检索到目标 chunk 后，按原始顺序检索其相邻的 N 个 chunk（一个滑动窗口）。

最佳实践：chunking 阶段追求精确定位，retrieval 阶段追求充分上下文。两者分工明确，互不干扰。

权衡：expanded chunks 消耗更多 context window，增加成本和延迟。需要根据具体场景调整扩展窗口的大小。

❓ 测验 6：什么时候最适合使用 chunk expansion？

  • ✓ 正确！chunk expansion 就是在检索阶段补充 chunking 阶段丢失的上下文。
  • ✓ 正确！这就是 chunk expansion 的设计初衷——不改 chunking 策略的前提下，靠检索时扩展窗口改善 LLM 理解。
  • ✗ chunk expansion 解决的是上下文缺失问题，不是 embedding 质量问题。

七、FDE 视角：不要过度工程化

作为 FDE，你面对的是一个真实世界的工程决策。记住以下五条原则：

  1. 从 fixed-size 开始——Pinecone 明确建议先从这里起步，只在证明不足时才升级
  2. Chunking 不是独立问题——必须与 embedding model 选择、query 类型、下游场景联合优化
  3. Eval 驱动调优——用代表性数据集测试多种 chunk size 和策略，比较 recall@K 和生成质量
  4. Contextual Retrieval 是当前 SOTA——知识库规模 > 200K tokens 且检索质量是关键瓶颈时优先尝试
  5. Chunk expansion 是低成本提升——不改 chunking 策略就能显著改善 LLM 理解
