模块二：AI 应用工程
第 07 课 · 从文档解析到生产部署的完整工程管线

RAG 企业AI落地 检索增强生成

📌 一句话定义

RAG 全链路 = 把非结构化数据变成 LLM 可消费的上下文，并在生产环境中保证权限、质量、延迟和成本的完整工程管线。

洞察一：RAG 不是"把文档丢进向量库"

最成功的 RAG/Agent 实现使用的是"简单、可组合的模式"，而非复杂框架。Anthropic 将 RAG 系统拆解为三个层次：

层次职责关键决策
Augmented LLM（增强层）在 LLM 之上叠加 retrieval、tools 和 memory每种能力的接口必须 well-documented
Workflow（编排层）用 prompt chaining、routing 等确定性模式编排已知子任务用确定性模式
Agent（自主层）LLM 自主决策检索策略子任务不可预测时才启用
关键决策

不要一开始就建 agent。先用 retrieval + prompt 优化单个 LLM 调用。只有在简单方案不够时才增加复杂度。正确顺序是：先定义好数据接口和检索接口，再考虑要不要用 workflow 或 agent 编排。

洞察二：Chunking 策略决定 RAG 上限

Chunking 不是预处理步骤，而是检索质量的第一道闸门。没有"最佳"chunking，只有"最适合场景"的 chunking。

策略适用场景不适用场景
Fixed-size（固定大小）简单文档、快速原型结构丰富的文档
Recursive（递归分割）Markdown/代码等有分隔符的文档段落间语义紧密的文档
Semantic（语义分割）需要保持语义完整性计算开销大
Document-aware（文档感知）PDF/HTML/Word 多格式需要自定义解析器
Agentic（智能体驱动）复杂查询、多跳推理延迟敏感场景
工程实践

在生产环境中，固定大小 chunk（500-1000 tokens）+ 重叠（10-20%）仍然是大多数场景的最佳起点。只有当基准效果不达标时，才投入工程资源探索更复杂的策略。

OpenAI 的 Retrieval API 展示了另一个关键维度：chunk 不仅是文本块，而是带有 attributes 的结构化单元。每个 chunk 可以携带 region、author、date 等元数据，这些元数据在检索时可以作为 filter。

洞察三：Hybrid Search 是生产标配

纯语义搜索在工业场景中不够用。混合检索（Hybrid Search）结合了两种互补信号：

信号类型捕捉什么代表算法
语义搜索意图和概念层面的相关性向量相似度
关键词搜索精确匹配和专有名词BM25 / lexical

生产级混合检索的标准流程：

Query → [Semantic Search + Keyword Search] → Reciprocal Rank Fusion → Top-K → Reranker → Final Context

三个关键设计点：

  1. RRF（倒数排名融合）在 reranker 之前合并两个检索器的结果，避免了单一检索器的盲区
  2. Reranker 是最后一道防线——Cohere 的 rerank-v3.5 模型专门针对 query-document 相关性训练，比 embedding 相似度更精确
  3. Reranker 的计算成本远低于 LLM——可以在毫秒级对 100+ 候选文档做精细排序
研究发现

在生产 RAG 系统中，引入 reranker 后，最终答案质量提升可达 15-30%。

洞察四：RAG 评估必须分层

传统 RAG 评估存在一个根本性问题：基于 query-document 相关性标签的检索评估与 RAG 系统下游性能的相关性很低。检索模型返回的文档"相关"，不代表 LLM 用它生成的答案就"好"。

评估层评估什么指标
Retrieval（检索层）检索到的文档是否相关Recall@K, MRR, NDCG
Generation（生成层）LLM 是否正确使用检索内容Faithfulness, Answer Relevance
End-to-End（端到端）用户是否得到正确答案Exact Match, BLEU, LLM-as-Judge
工程实践

在 FDE 项目中，建议先投入 80% 精力在 retrieval 层评估，因为这是 RAG 系统的瓶颈所在。大多数 RAG 失败案例的根因是"没找到正确的文档"，而非"LLM 没用好找到的文档"。

洞察五：Permission-Aware Retrieval

在企业客户场景中，RAG 面临的最大工程挑战往往不是"怎么检索得更准"，而是"怎么确保用户只能检索到他有权访问的内容"。

权限过滤有两种模式：

模式时机优势
Early Filter（早期过滤）检索时通过 metadata filter 限制范围安全性高、效率高、可审计
Post-Generate（后置过滤）生成后再做权限校验实现简单但风险大
关键设计原则

权限模型必须在数据摄入阶段就嵌入到 chunk 的元数据中，而不是在检索阶段动态计算。这需要在数据管道设计中提前规划 RBAC/ABAC 模型。
