模块二：AI 应用工程

第 12 课 — RAG Eval 与幻觉分析

从指标设计到落地实践：如何判断你的 RAG 系统到底做得好不好

📌 核心问题：怎么知道 RAG 系统"做得好"？

RAG 系统的评估不是单一指标问题，而是需要分层度量。检索层、生成层、端到链路层各有一套指标，而幻觉检测则是贯穿全链路的诊断工具。

FDE 在现场部署时，必须建立可量化的 "what good looks like" 框架，让 AI 系统持续可衡量、可改进。没有评估，就没有改进。

1. RAG 评估的三层架构

第一层：检索质量（Retrieval Quality）

检索层回答的问题是："找对材料了吗？"

  • Context Precision（查准率）：检索到的文档中有多少是真正相关的？—— 少而精
  • Context Recall（查全率）：所有相关文档中有多少被检索到了？—— 精而全

传统评估方法（基于查询-文档相关性标签）与 RAG 系统下游性能相关性极低。eRAG 方法提出用 LLM 逐个评估每篇文档对最终回答的贡献，通过下游任务的 ground truth 反向推导文档相关性，Kendall's τ 相关性比基线提升 0.168–0.494，同时 GPU 内存消耗降低 50 倍。

第二层：生成质量（Generation Quality）

生成层回答的问题是："答对了吗？答得靠谱吗？"

  • Faithfulness（忠实度）：生成的答案是否严格基于提供的上下文？这是检测幻觉的核心指标
  • Answer Relevancy（答案相关性）：答案是否直接回答了用户的问题？

第三层：端到端评估

当检索和生成各自表现良好但整体效果差时，问题通常出在 prompt 设计或系统编排上。

关键方法：ablation study——分别关闭检索模块（纯 LLM 回答）和关闭生成模块（仅返回检索片段），才能定位瓶颈在哪一层。

🧠 测验 1：某 RAG 系统检索查准率 92%、查全率 88%，生成忠实度 95%，但端到端评分很低。最可能的瓶颈在哪里？
🧠 测验 2：做 ablation study 时，关闭检索模块后纯 LLM 的回答反而更好，这说明什么？

2. 幻觉的四种类型与检测策略

幻觉不是单一现象，而是四种不同类型的组合。每种类型需要不同的检测策略：

幻觉类型描述检测方式
上下文无关幻觉 上下文中已有正确答案，模型仍基于预训练知识生成矛盾信息 Faithfulness 指标 + 逐句比对
上下文缺失幻觉 检索器未召回关键文档，模型"被迫"编造答案 Context Recall + 人工标注 golden set
跨文档冲突幻觉 多个检索文档存在矛盾，模型选了错误的一个 Multi-hop 评估，检查权威来源识别
推理幻觉 上下文正确、引用正确，但逻辑推理链条出错 Chain-of-Thought 级别评估
FDE 最常遇到的问题：上下文缺失幻觉。客户数据质量差、权限过滤过度、chunking 策略不当都会导致召回率低，模型被迫编造答案。
🧠 测验 3：一个 RAG 系统在客户现场频繁生成看似合理但与事实不符的答案，经排查发现是因为权限过滤过于严格导致关键文档未被召回。这属于哪种幻觉？

3. Golden Set：评估的上限由它决定

几乎所有 RAG 评估框架都依赖 golden set（黄金测试集），但其构建质量被严重低估。

构建要点：

  • 规模：至少 100–200 条高质量样本。生产系统覆盖 5–10 个典型用户场景，每个场景 20+ 样本
  • 多维标注：每条样本应包含 query、golden_answer、retrieved_contexts（实际检索结果，非理想结果）、ground_truth_documents
  • 分布代表性：必须覆盖真实用户查询的分布——简单事实查询、多跳推理、模糊意图、对抗性提问等
常见错误：只用简单问题测试系统，导致上线后遇到复杂查询时性能断崖式下降。

最新趋势是用 LLM 自动生成 golden set，但需注意合成数据可能存在系统性偏差——用更强模型生成评估数据时，弱模型在评估中的得分会系统性偏低（评估器偏差问题）。

4. 线上 vs 线下评估：FDE 的关键挑战

离线评估（offline eval）和线上评估（online eval）之间存在巨大差距，这是 FDE 在客户现场最常遇到的工程挑战。

维度Offline EvalOnline Eval
环境 隔离环境，固定测试集 生产环境，真实用户交互
优点 快速、低成本、可复现 捕捉真实交互模式、反馈循环
缺点 无法模拟真实用户行为 成本高、不可控变量多
对齐指标 技术指标（Precision、Recall） 业务指标（时间节省率、错误减少率）

A/B Testing 的特殊陷阱

在 AI 系统中做 A/B testing 比传统软件更复杂。LLM 输出是非确定性的，同一 query 两次调用可能得到不同结果。解决方案：

  • 使用固定 seed 的确定性评估
  • 在统计显著性检验时增加样本量

反馈闭环

最有效的线上评估是将用户反馈自动转化为新的 golden set 样本，形成持续改进循环。OpenAI 的成功指标明确要求："eval-driven feedback must change product and model roadmap"——评估不能只是报告，必须驱动产品决策。

5. 成本-精度权衡：FDE 的工程决策核心

评估体系本身有成本，FDE 需要在评估深度和部署速度之间做出权衡。

评估级别成本精度适用阶段
规则匹配（Exact Match / BLEU） 极低 低 早期原型验证
Embedding 相似度（Cosine / BERTScore） 低 中 迭代开发期
LLM-as-a-Judge 中 高 上线前验收
人工评审（Human Evaluation） 极高 最高 关键业务场景

推荐做法：开发阶段用 embedding 相似度做快速迭代 → 上线前用 LLM-as-a-Judge 做全面评估 → 关键业务场景抽样人工评审。

🎯 本课总结

  • 三层评估：检索层（Precision/Recall）、生成层（Faithfulness/Relevancy）、端到端（ablation study 定位瓶颈）
  • 四类幻觉：上下文无关、上下文缺失、跨文档冲突、推理幻觉——每种需要不同检测策略
  • Golden Set 是关键：质量决定评估上限，必须覆盖真实查询分布，注意评估器偏差
  • 线上+线下结合：离线做快速迭代，线上对齐业务指标，形成反馈闭环驱动产品决策
  • 成本-精度权衡：按阶段选择合适评估级别，关键场景抽样人工评审
