第 11 课：Permission-aware Retrieval — 权限感知检索

在向量检索中嵌入 RBAC/ABAC 权限模型，确保用户只能访问其有权获取的内容
RAG RBAC ABAC 向量数据库 数据安全

一、为什么权限感知检索是企业的红线

想象一下：你在一家金融公司部署了一个 RAG 系统，员工可以用自然语言查询内部文档。某天，一个初级分析师竟然通过系统检索到了 CEO 的薪酬方案——那份标注为「仅限高管」的文档。

这就是数据越权泄露。它不是理论风险，而是企业客户拒绝 AI 系统上线的头号安全顾虑。权限感知检索的核心目标很简单：在向量检索的每一层嵌入权限控制，确保用户只能看到自己有权访问的内容。

核心原则：权限过滤不是事后补丁，而是从第一天就嵌入到索引和查询两个阶段的基础设施。

二、两大过滤时机：索引时 vs 查询时

权限过滤在 RAG 中有两个关键切入点，各有优劣：

索引时过滤（Index-time Filtering）

在文档入库时，为每个 chunk 打上权限标签（如 allowed_roles: ["engineering"]），存入向量数据库的 metadata 字段。

优点：实现简单，查询时无需额外处理。
缺点：权限动态变化（如用户角色变更）时需要重新索引所有文档，成本极高。

查询时过滤（Query-time Filtering）

在检索时将用户权限作为 metadata filter 实时传递给向量数据库。

优点：权限变更即时生效，无需重新索引。
缺点：每个查询都带 filter，可能影响检索性能。

最佳实践：两者结合使用——静态权限（如部门归属）用索引时过滤，动态权限（如临时项目授权）用查询时过滤。

三、RBAC vs ABAC：两种权限模型

企业环境中最常见的两种权限模型——RBAC 和 ABAC——在 RAG 中有不同的实现路径：

RBAC

基于角色的访问控制

将权限抽象为角色（如 "manager"、"engineer"），文档标注允许的角色列表。

适用场景：大多数企业，结构简单，易于管理。

示例：allowed_roles: ["engineer", "manager"]

ABAC

基于属性的访问控制

基于属性组合判断：department == "finance" AND clearance_level >= 3 AND location == "US"

适用场景：政府、金融等高合规要求场景，需要细粒度控制。

FDE 实战要点：Google Cloud 的 FDE 岗位要求理解 RBAC 和 ABAC 的应用场景，并能根据客户需求选择合适的模型。不要盲目追求 ABAC 的精细度——简单场景用 RBAC 就够了。

四、三大架构设计模式

模式 A：推荐起步

Metadata Filter 模式

查询时构建 metadata filter 并传给向量数据库。这是最简单、最通用的方案，适合大多数起步场景。

{
  "$and": [
    {"$or": [{"allowed_roles": {"$in": ["engineer", "manager"]}}]},
    {"department": {"$eq": "engineering"}}
  ]
}
模式 B：简单但不可靠

权限感知的 Embedding 模式

将用户权限注入 prompt："你是 engineering 部门的工程师，仅使用你有权限访问的文档回答问题。"

注意：LLM 不保证遵守权限约束，仅适用于低风险场景。

模式 C：高性能方案

权限预计算 + 缓存模式

为每个用户预计算可访问的文档 ID 集合，存储在 Redis 等缓存层。检索时先查缓存再执行向量搜索。

适用场景：高并发、权限稳定的企业环境。

五、向量数据库的性能挑战

metadata filtering 在不同向量数据库上的表现差异很大。作为 FDE，你需要了解这些权衡：

  • Pinecone：支持 indexed metadata filtering，但 filter 数量有上限（通常每索引 50 个 unique values）。
  • Weaviate：原生支持 hybrid search + filter，性能较好。
  • Qdrant：支持 pre-filtering 和 post-filtering 两种模式，pre-filtering 性能更优。
  • Milvus：支持 scalar + vector 联合查询，适合大规模权限过滤。

关键权衡：filter 的 cardinality（基数）越高，性能下降越明显。当权限维度超过 50 个 unique values 时，需考虑分层索引或预计算权限矩阵。

六、FDE 实战 Checklist

在实际项目中落地权限感知检索，按以下步骤推进：

  • 确定客户使用的权限模型（RBAC / ABAC / 混合）
  • 选择向量数据库并验证其 metadata filter 性能
  • 设计 chunk-level 权限标注方案
  • 实现查询时的权限 filter 注入
  • 编写 eval case 测试越权场景（用户 A 不应看到用户 B 的文档）
  • 监控越权率指标
  • 设计权限变更后的同步机制

重要提醒：Chunking 策略需要考虑权限边界——一个 chunk 不应跨越不同的权限级别。否则即使做了权限过滤，也可能因为 chunk 粒度太粗而导致信息泄露。
