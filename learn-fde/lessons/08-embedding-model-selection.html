Embedding Model 选择

🧠 Embedding Model 是 RAG 系统的"翻译层"——它决定了用户查询和文档能否在同一个语义空间中被匹配。选错模型等于在起跑线上输了召回率。

一、没有"最好"，只有"最适合"

MTEB（Massive Text Embedding Benchmark）是目前最权威的 Embedding 评估基准，覆盖 100+ 任务、110+ 语言。选型时最关键的三个维度：

短文本 vs 长文本
text-embedding-3-small 在短文本检索上表现优异；但 C-Pack（BAAI）的 long-context 模型在 8K+ token 文档检索上显著领先。
多语言需求
bge-m3 同时支持多语言、多粒度（段落/句子/关键词）嵌入，是全球化产品的首选开源方案。
领域适配
通用模型在垂直领域（法律、医疗、金融）往往不如微调后的专用模型。例如 bge-large-zh-v1.5 在中文任务上远超通用模型。

FDE 洞察：在 Discovery 阶段必须明确客户数据的语言构成和领域分布，不能直接套用通用模型。

二、维度选择：精度、成本、速度的三角博弈

Embedding 模型的维度直接决定了存储成本、检索速度和最终精度。决策框架如下：

维度范围典型模型适用场景成本影响
256–512text-embedding-3-small, bge-base快速原型、低预算项目最低
768–1024bge-large, jina-embeddings-v3生产级 RAG 平衡点中等
3072+text-embedding-3-large, Cohere embed-multilingual-v3高精度要求、复杂检索最高
四步决策法：
① 用低成本模型做 baseline → ② 建立 golden set 后对比 2–3 个候选 → ③ 若召回率差距 > 5%，考虑升级或引入 reranking → ④ 若超预算，考虑量化（FP16→INT8）或蒸馏。

FDE 洞察：FDE 不应只做"模型选择"，而应建立"模型选择→评估→部署→监控"的完整流程。

三、Open Source vs Proprietary：FDE 的实际考量

在客户现场，合规约束往往比模型性能差异更重要。

维度Open Source（BGE, Jina, Cohere API）Proprietary（OpenAI, Google）
部署灵活性可 self-host，满足数据不出域必须调用 API，受限于供应商
成本控制无 per-token 费用，GPU 成本固定按 token 计费，大规模时成本高
安全合规完全可控，适合政府/金融需确认供应商合规认证
性能天花板顶级模型接近专有模型通常有轻微领先
💡 Palantir 的经验：在政府和高安全场景中，self-hosted embedding 是刚需。OpenAI Gov FDE 岗位明确要求处理"数据不出域"约束。FDE 必须在早期就做出部署架构决策。

四、双阶段架构：Embedding + Reranking 生产标配

单一 embedding 模型无法同时优化召回率和排序精度。行业标准架构：

Query → Embedding Model → Vector DB (Top-K=100) → Reranker (Top-K=10) → LLM

主流 reranker 模型对比：

  • Cohere Rerank：多语言支持好，API 简单，延迟 ~50ms
  • BGE-Reranker：开源，可 self-host，中文效果好
  • Jina Reranker v2：支持长上下文，多语言
  • E5-Reranker：微软出品，与 E5 embedding 生态一致
ROI 分析：reranking 通常在 2–5% 的召回率提升上投入 10–20% 的计算成本，性价比极高。FDE 应将其作为默认选项而非可选组件。

五、Embedding 选型是持续迭代，不是一次决定

MTEB 基准每季度更新，新模型不断涌现。FDE 需要建立的体系化能力：

  • 定期评估机制：每季度用最新模型跑一遍 golden set
  • A/B 测试框架：在生产环境中并行运行多个 embedding 模型
  • 监控仪表板：跟踪平均相似度分布、冷启动查询的召回率变化
  • 降级预案：当新模型效果不稳定时，能快速回滚到已知稳定版本
💡 Embedding 选型是 FDE 产品化思维的重要体现——从"选择一个模型"升级为"构建一个模型治理体系"。

六、实战检查清单

在客户现场选择 Embedding Model 时的标准流程：

  1. 明确约束：语言、领域、数据敏感度、预算、延迟要求
  2. 选定候选：2–3 个模型（1 个开源 + 1 个专有 + 1 个备选）
  3. 构建 golden set：至少 50–100 个真实查询-文档对
  4. 批量评估：计算 Recall@K、NDCG、MRR
  5. 成本测算：按预计日请求量计算月度成本
  6. PoC 验证：在客户环境中跑 1 周真实流量
  7. 监控上线：部署后持续跟踪质量指标
