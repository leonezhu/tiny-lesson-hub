第 10 课 — Hybrid Search 与 Reranking

BM25 与向量检索的融合策略、Cross-Encoder 重排序架构及生产级实践

Hybrid Search + Reranking 深度研究

关键词匹配与语义理解的融合之道

一、为什么需要 Hybrid Search？

纯向量检索（Dense Retrieval）在语义相似性上表现优异，但它有一个固有缺陷：对精确关键词不敏感。当用户查询某个产品型号、代码函数名或具体数值时，向量嵌入可能无法准确捕捉这些细节。

反过来，纯 BM25 擅长关键词精确匹配，却无法理解语义相关性——比如用户搜索"银行开户流程"，BM25 可能找不到文档中写"金融机构注册步骤"的相关内容。

核心洞察：Hybrid Search 的目标不是选一个更好的单一模型，而是将 BM25 和向量检索的优势结合，在召回阶段兼顾语义和关键词。

具体而言，BM25 的核心优势包括：

  • 对精确关键词匹配天然敏感（产品型号、函数名、数值）
  • 不需要向量索引，存储开销极低
  • 可解释性强——词频（TF）和逆文档频率（IDF）都有明确含义
  • 在多语言场景下表现稳定

二、两阶段检索架构：生产级 RAG 的标准范式

现代检索系统普遍采用 Retriever → Reranker 的两阶段架构：

  1. 召回阶段（Retrieval）：用轻量级模型（dense embedding 或 sparse BM25）从百万级文档中快速召回 Top-K（通常 50–200 篇）。追求速度。
  2. 排序阶段（Reranking）：用计算成本更高的 Cross-Encoder 对候选集进行精细排序，最终输出 Top-N（通常 3–10 篇）。追求精度。

这种分层设计直接决定了系统的延迟预算和成本结构——召回阶段可以并行处理大量文档，而排序阶段虽然慢，但只需要处理少量候选。

检索流程：Dataset → Embedding → Indexing → Retrieval → Reranking → Evaluation

BGE 官方教程将检索流程明确分为这 6 步，每一层都可以独立优化。对于 FDE 来说，理解这种架构分离至关重要，因为它意味着你可以在不同阶段投入不同的计算资源。

三、Cross-Encoder Reranker：精度与成本的权衡

Cross-Encoder 需要对 query 和每篇文档做联合编码，时间复杂度为 O(K × N)——K 是候选数，N 是文档长度。相比 Bi-Encoder（单次编码 query 和所有文档），Cross-Encoder 慢 10 到 100 倍。

但代价换来的是显著的质量提升：

指标Bi-EncoderCross-Encoder
推理速度快（一次编码）慢（逐对计算）
nDCG@10 提升基准+10% ~ +30%
存储开销低中（需候选集）

BGE 提供了完整的 reranker 产品线，体现了业界的一个共识：reranker 质量与参数量正相关，但推理成本也急剧上升。

  • 轻量 bge-reranker-v2-m3（568M 参数）— 适合低延迟场景
  • 中等 bge-reranker-v2-minicpm-layerwise — Layer-wise 设计允许选择中间层加速推理
  • 高精度 bge-reranker-v2.5-gemma2-lightweight（9.24B 参数）— 适合高精度场景

四、Query Rewriting：检索的前置关键步骤

在检索之前对 query 进行改写，可以显著提升后续检索效果。这不是可选功能，而是生产级 RAG 系统的标准配置。

三种核心改写策略

  • Query Expansion（扩展）：用同义词扩展查询。例如 "bank" → "bank financial institution river"，覆盖更多语义维度。
  • Query Decomposition（分解）：将复杂查询拆分为子查询。例如 "比较 GPT-4 和 Claude 的上下文窗口大小" → ["GPT-4 上下文窗口", "Claude 上下文窗口"]，分别检索后再合并结果。
  • Step-back Prompting（后退）：先生成一个更泛化的查询来获取背景知识，再基于背景知识做细粒度检索。
FDE 注意：LangChain 和 LlamaIndex 都内置了 query rewriting 组件。在设计 RAG 系统时，应将其作为标准配置而非可选功能。

五、BGE-M3：一个模型覆盖多种检索策略

BAAI 的 bge-m3 是一个里程碑式的作品——它在一个模型中实现了五种能力：

  • Dense Retrieval：传统的双编码器向量搜索
  • Sparse Retrieval：BM25-like 的词级匹配
  • Multi-Vector (ColBERT)：token-level 的细粒度匹配
  • Multi-Linguality：支持 100+ 语言
  • Multi-Granularity：支持最高 8192 tokens 的文档

这种统一架构的优势在于：

  • 单一模型管理，降低运维复杂度
  • 可以针对不同查询类型动态选择检索策略
  • 训练数据共享，各模式互相增强
FDE 实战启示：在大多数企业场景中，一个 bge-m3 可能就足够了——不需要单独维护 BM25 索引 + 向量索引 + reranker 三个独立系统。

六、FDE 实战建议：渐进式部署路径

在生产环境中部署 Hybrid Search + Reranking 时，建议遵循以下原则：

  1. 召回阶段：优先使用 bge-m3 的多功能检索能力，根据查询类型自动选择 dense/sparse/multi-vector 策略
  2. 排序阶段：根据延迟预算选择 reranker 模型——低延迟场景用 v2-m3（568M），高精度场景用 v2-gemma（2.51B）
  3. 评估闭环：始终维护 golden dataset，定期评估 reranking 前后的 Recall/MRR/nDCG 变化
  4. 成本监控：记录每次请求的 tokens/cost/latency，特别是 reranker 阶段的额外开销
  5. 渐进式部署：先上线 dense-only 检索，验证效果后再加入 BM25 hybrid 和 reranker
关键提醒：不要一开始就堆砌所有组件。先从 dense-only 开始，建立基线，然后逐步加入 BM25 hybrid 和 reranker，每一步都通过 eval harness 量化验证效果。
