AI Agent——让 AI 从"回答"走向"行动"

Lesson 9 · ~10 minutes · AI 前沿技术

本节课目标，你将理解 AI Agent 的核心架构——为什么它不只是"更强的提示词"，而是一种全新的软件范式。学完后你能解释 Agent Loop 的四步循环，区分 Agent 与传统 AI 应用的不同，并了解主流 Agent 框架。

为什么需要 AI Agent？

在第 7 课和第 8 课中，我们学习了 LLM 和 RAG。但无论是聊天机器人还是知识库问答，它们本质上都是"你问它答"——被动地接收指令、返回文本。然而现实中很多任务不是问答，而是需要自主决策、连续行动、使用工具。

比如：你让 LLM 写一份报告，它只会返回文本。但如果你让它"调研三家竞争对手的产品定价，生成对比表格并保存到指定文件夹"——这就需要一个能搜索网页、调用 API、编写代码、写入文件的系统。这就是 AI Agent。

核心区别：传统 AI = 输入文本，输出文本。AI Agent = 感知环境 → 推理规划 → 执行动作 → 反思调整，形成一个自主闭环。

AI Agent 的四步循环（Agent Loop）

Agent 的核心是一个持续运行的循环，它让 AI 从"单次对话"变成"自主执行任务"。这个循环可以简化为四个步骤：

1

感知（Observe）

Agent 接收来自环境的输入——用户的请求、当前系统状态、之前行动的反馈。就像人用眼睛和耳朵收集信息一样，Agent 通过 API、文件读取、数据库查询等方式感知世界。

2

思考（Think / Planning）

Agent 利用 LLM 进行推理：分析当前状态、制定行动计划、决定下一步该做什么。这一步是 Agent 的"大脑"——它需要理解用户意图、评估可用工具、做出决策。常用的推理模式包括 ReAct（推理+行动交替）、Plan-and-Execute（先规划再执行）等。

3

行动（Act）

Agent 执行具体的动作——调用 API、运行代码、搜索互联网、发送消息、修改文件。关键是这些动作是动态选择的，不是硬编码的。Agent 根据当前需求自动决定使用哪些工具。

4

反思（Reflect）

Agent 检查结果是否达到预期。如果没有——调整策略，重新规划，继续循环。如果达到了——终止循环，返回结果。这个反思能力让 Agent 能在出错时自我修正，而不是像传统程序那样直接崩溃。

与传统 AI 应用的对比

理解 Agent 的关键是对比它和传统系统的差异：

传统 AI 应用（聊天机器人 / RAG）

  • → 流程：用户输入 → LLM 生成 → 返回文本 → 结束
  • → 控制流：一次请求，一次响应，没有循环
  • → 工具使用：通常没有，或仅静态配置
  • → 错误处理：无法自我修正，用户需要重新提问

AI Agent

  • → 流程：感知 → 规划 → 行动 → 反思 →（循环）→ 完成
  • → 控制流：多轮循环，每一步的输出影响下一步
  • → 工具使用：动态选择工具，按需组合
  • → 错误处理：反思后自我修正，多轮尝试直到成功

Agent 的关键组件

一个完整的 Agent 系统需要以下组件协同工作：

大脑（LLM）

LLM 是 Agent 的推理核心，负责理解意图、制定计划、做出决策。选择哪个模型很重要——长上下文、强推理能力的模型（如 Claude、GPT-4）更适合做 Agent 的大脑。

工具箱（Tools）

Agent 需要能调用的外部能力，例如：搜索、计算器、代码执行、数据库查询、API 调用、文件读写。工具越多、越灵活，Agent 能完成的任务就越复杂。

记忆（Memory）

Agent 需要记住之前的步骤和结果。短期记忆通常用对话历史实现，长期记忆可以借助向量数据库（RAG）存储知识。好的记忆机制让 Agent 在多轮对话中保持一致性。

规划器（Planner）

决定任务分解策略。简单任务可能不需要规划，但复杂任务需要 Agent 先将大目标拆成子任务，再逐一执行。规划能力决定了 Agent 处理复杂任务的成功率。

一个现实例子：你让 Agent "帮我对比三家公司的股价并写一份简短分析"。Agent 需要：① 搜索这三家公司的股票代码 ② 调用金融 API 获取股价数据 ③ 计算涨幅 ④ 搜索相关新闻 ⑤ 综合分析 ⑥ 生成报告。每一步都需要 Agent 自主选择工具和判断，而不是预先写好的流程。

两种典型的 Agent 模式

模式一：ReAct（推理 + 行动交替）

ReAct 是目前最常用的 Agent 模式——"Thought-Action-Observation"循环。Agent 在每个步骤中交替进行思考和行动： 

Thought: 我需要先找到这三家公司的股票代码
Action: 搜索("腾讯 阿里巴巴 美团 股票代码")
Observation: 腾讯(0700.HK)、阿里巴巴(9988.HK)、美团(3690.HK)

Thought: 现在获取最新的股价数据
Action: 调用金融API("0700.HK", "9988.HK", "3690.HK")
Observation: 腾讯 380 HKD, 阿里巴巴 75 HKD, 美团 110 HKD

Thought: 还需要搜索近期新闻...
Action: 搜索("腾讯 阿里巴巴 美团 近期新闻")
...（继续循环直到完成）

模式二：Plan-and-Execute（先规划，后执行）

Agent 先用 LLM 制定完整计划，然后逐步执行。这种方式适合任务明确、步骤可预测的场景： 

Plan:
1. 搜索三家公司基本信息
2. 获取最近一个月的股价数据
3. 计算涨跌幅和波动率
4. 搜索近期相关新闻
5. 综合对比并生成报告
6. 保存到 ~/Documents/stock_analysis.md

Step 1: 搜索... [完成]
Step 2: 获取数据... [完成]
Step 3: 计算... [完成]
...（按计划执行，如需调整再回退到规划）
模式选择：ReAct 更灵活，适合开放式任务（"帮我做一个报告"）；Plan-and-Execute 更可控，适合步骤清晰的复杂任务（"部署这个应用到服务器"）。实际系统中经常混合使用两种模式。

主流 Agent 框架

目前主流的 Agent 框架包括：

LangChain / LangGraph

Python/JS 生态中最流行的 Agent 框架。LangChain 提供了完整的工具链和抽象层，LangGraph 则引入了状态机和有向图的概念，让 Agent 的流程可以被可视化编排。适合构建生产级的 Agent 系统。

AutoGPT / BabyAGI

早期的自主 Agent 项目，展示了 LLM 如何自行分解任务、管理待办列表、持续执行。它们的价值在于理念——证明了 Agent 的可行性，虽然实际稳定性和效率仍需改进。

OpenAI Function Calling

OpenAI 官方提供的结构化输出接口，让 LLM 可以直接"调用函数"而非生成文本。虽然不是完整的 Agent 框架，但为构建轻量级 Agent 提供了最基础的支撑。

Agent 的挑战与局限

尽管 Agent 前景广阔，但当前仍面临几个核心挑战：

  • ! 无限循环：Agent 可能在错误的路径上反复尝试，消耗大量 token 和成本。必须设置最大迭代次数和超时限制。
  • ! 成本失控：每多一轮循环就多一次 API 调用。复杂任务可能需要几十轮推理，成本可能超过直接人工处理。
  • ! 幻觉风险：Agent 可能"发明"不存在的 API 或工具，导致执行失败。幻觉在工具调用场景中后果更严重。
  • ! 调试困难：多轮循环的执行路径难以追踪，问题可能出现在任意一轮，不像传统代码那样容易调试。
现实认知：Agent 不是万能药。目前最适合的场景是"目标明确但路径不确定的任务"——比如"调研某话题"、"分析某问题"。对于步骤固定的任务，传统自动化（脚本、工作流引擎）反而更可靠、更便宜。
