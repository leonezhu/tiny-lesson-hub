AI 应用架构：多模态、Agent 系统与企业部署

第十一课 · ~12 分钟 · 从理论到实践的桥梁

本课目标，你将理解 AI 应用如何从"一个模型"变成"一套系统"——多模态输入输出、Agent 自主决策、企业级部署架构。学完后能够画出典型 AI 应用的架构图，并知道每个组件负责什么。

为什么需要"架构"？

前面我们学了神经网络、Transformer、LLM、RAG、Agent、RLHF——每一课都在讲一个核心技术。但在现实中，没有人只把模型直接给用户用。一个真正可用的 AI 应用，需要把模型包装在一套完整的架构里。

想象一下：你想做一个"智能客服"。光有一个 LLM 是不够的——你需要让它能访问你的产品数据库（RAG）、能调用工单系统（工具使用）、能处理图片（多模态）、能在高并发下稳定运行（企业部署）。这就是 AI 应用架构要解决的问题。

一、多模态架构

早期 AI 应用主要处理文本。但现实世界是多模态的：人有声音、图像、视频、传感器数据。多模态 AI 架构的目标是让模型能够同时理解和生成多种类型的数据。

多模态架构的核心组件

一个典型的多模态系统包含以下层次：

  • 输入编码器（Encoder）：将文本、图像、音频分别转换为向量表示。例如 CLIP 用两个独立的编码器处理图片和文本，然后在同一个向量空间中对齐它们。
  • 融合层（Fusion Layer）：将不同模态的向量合并。可以是早期融合（在输入端合并）、晚期融合（在输出端合并），或交叉注意力融合（让一个模态的 token 去"关注"另一个模态的信息）。
  • 统一模型（Unified Model）：在融合后的表示上进行推理。例如 GPT-4V 可以同时看到图片内容和文本指令，并生成文本回答。
  • 输出解码器（Decoder）：将模型的内部表示转换回人类可读的形式——可能是文本、图像或音频。
关键理解：多模态不是简单地把多个模型拼在一起。真正的多模态意味着不同模态之间共享语义空间——模型理解"猫的照片"和"猫"这个词指向同一个概念。这种对齐是通过大规模跨模态训练实现的。

二、Agent 系统架构

Agent 是 AI 应用中更高级的形式：它不只是回答问题，而是自主规划、调用工具、执行任务。一个 Agent 系统通常包含以下组件：

👤 用户（自然语言交互界面）
↓ ↑
🧠 Agent 核心
├── 规划器（Planner）：将大任务拆解为子步骤 ├── 记忆系统（Memory）：短期上下文 + 长期知识库 ├── 工具调度器（Tool Router）：决定调用哪个外部工具 └── 反思模块（Reflector）：检查结果，必要时重新规划
↓ ↑
🤖 大语言模型（推理引擎）
↓ ↑
🔧 工具层：API、数据库、搜索引擎、代码执行环境

Agent 的典型工作流程

  • 接收任务：用户说"帮我调研竞品并写一份报告"
  • 规划分解：Agent 拆分为：①搜索竞品信息 ②整理对比表格 ③撰写分析报告
  • 工具调用：调用搜索引擎获取数据，调用代码解释器生成表格
  • 结果整合：将收集到的信息整合成结构化报告
  • 反思迭代：检查报告是否完整，是否需要补充更多数据
常见误区：很多人认为 Agent = 聊天机器人 + 一些工具。实际上，Agent 的核心特征是闭环决策——它能根据中间结果动态调整下一步行动，而不只是按预设流程走。这是"工具调用"和"自主 Agent"的本质区别。

三、企业级部署架构

把模型跑起来和在企业中稳定运行之间，隔着巨大的工程鸿沟。企业级 AI 部署需要考虑以下维度：

维度 实验环境 企业级部署
延迟 秒级可接受 通常需要 <200ms（实时交互）或 <5s（批量任务）
吞吐量 每秒几次请求 数千到数万 QPS，需要负载均衡和批处理
可靠性 偶尔出错没关系 需要降级策略、熔断机制、自动重试
安全合规 不需要 数据加密、访问控制、审计日志、GDPR 合规
成本 不计较 需要精细的成本追踪和优化（缓存、蒸馏、小模型路由）
监控 看日志就行 需要完整的可观测性：指标、日志、追踪、告警

企业 AI 部署的常见架构模式

  • 模型即服务（MaaS）：通过 API 调用云端模型（如 OpenAI API、Claude API）。最快上线，但数据出域，成本高。
  • 私有化部署：将模型部署在自己的服务器上。数据不出域，可控性强，但需要 GPU 基础设施和运维团队。
  • 混合架构：简单任务用云端 API，敏感任务用本地模型。兼顾成本和安全性，是目前主流选择。
  • 模型路由（Router）：根据任务复杂度自动选择模型——简单问题用小模型（便宜快），复杂问题用大模型（质量好）。

四、典型 AI 应用架构案例

案例：智能文档处理系统

某银行用它来处理贷款申请材料的审核

  • 输入：用户上传 PDF 扫描件（身份证、收入证明、银行流水）
  • OCR + 多模态解析：将扫描件转为结构化文本，提取关键字段
  • LLM 理解：用 LLM 理解文档内容，判断是否符合贷款条件
  • Agent 工作流：如果材料不全，自动生成补件通知；如果信息矛盾，触发人工复核
  • 决策输出：生成审核意见，写入银行核心系统
  • 合规层：所有操作留痕，满足金融监管要求
核心洞察：在这个案例中，没有任何一个单独的技术解决了问题。OCR 解决图像转文本，LLM 解决理解，Agent 解决工作流，合规系统解决监管。把它们有机组合在一起，才是 AI 应用的真正价值所在。

关键概念速记

  • 多模态：模型同时处理文本、图像、音频等多种数据类型
  • Agent 闭环：感知 → 规划 → 行动 → 观察 → 调整的循环
  • 工具调用（Function Calling）：让模型决定何时以及如何调用外部 API
  • 模型路由：根据任务自动选择最合适的模型
  • MaaS：Model as a Service，通过 API 使用模型
  • 私有化部署：模型跑在自己的基础设施上
