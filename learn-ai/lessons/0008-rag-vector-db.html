RAG 与向量数据库

Lesson 8 · ~10 minutes · RAG & 向量检索

通过本节课，你将理解 RAG（检索增强生成）为什么出现、它如何工作，以及为什么它是企业级 AI 应用的事实标准。学完后你能向别人解释"为什么大模型需要外挂知识库"。

大模型的"记忆困境"

大语言模型有一个根本性的矛盾：它知道很多，但它不知道"现在"的事。

GPT-4 的训练数据截止于 2023 年初，Qwen 或 Claude 的训练数据也有各自的截止日期。这意味着它们无法回答此后发生的事件——不管那个事件有多重要。此外，模型在参数中"记住"的知识是静态的，一旦训练完成就不再更新。重新训练整个模型的代价极其高昂，而且每次重训又会丢失旧知识。

更麻烦的是幻觉问题：当模型不知道答案时，它会自信地编造一个。2023 年的一项研究发现，即使在微调后，大模型在事实性问答中的幻觉率仍然高达 20-30%。

RAG 的核心思路

RAG（Retrieval-Augmented Generation，检索增强生成）的思路很简单：在回答问题之前，先翻一翻参考资料。

传统 LLM 的提问流程是：用户提问 → 模型凭"记忆"回答。RAG 的流程是：用户提问 → 先检索外部知识库 → 将检索结果作为上下文 → 模型基于参考资料回答。

这种方法由 Facebook AI 在 2020 年首次提出。它的核心贡献在于：把知识从"模型参数"转移到"外部可更新的知识库"，从而同时解决了知识过时和幻觉两个问题。

类比理解：硬盘 vs 内存

RAG 中的向量数据库相当于外部硬盘——容量近乎无限，存储成本低，可以随时更新。LLM 的模型参数相当于内存——容量有限（上下文窗口大小受限），推理成本高昂（Token 越多越贵），但处理速度快。RAG 的价值在于让模型在做重要决策前先"检索"相关记忆，而不是完全依赖直觉。

RAG 的三个步骤

1

索引（Indexing）

将外部文档切分成小块（chunk），每块 256-1024 个 token。然后用嵌入模型（embedding model）将每个块转换为高维向量，存入向量数据库（如 Pinecone、Weaviate、Chroma）。这个过程还建立从原始文档到向量索引的反向映射，确保最终回答可以追溯信息来源。

2

检索（Retrieval）

用户提出问题后，先用同一个嵌入模型将查询转换为向量，然后在向量数据库中进行相似性搜索，通常返回最相关的 K 个片段（K 取值 3-10，视场景而定）。向量搜索的本质是找到语义上最接近的文档片段，而不是关键词匹配。

3

生成（Generation）

将用户问题、检索到的文档片段和一个系统提示（如"基于以下信息回答问题"）一起发送给 LLM。模型被约束在检索到的知识范围内生成回答。这一步的输入格式大致是：[系统提示] + [检索文档 1] + [检索文档 2] + ... + [用户问题] → 模型输出。

关键设计决策

决策点 选项 影响
Chunk 大小 256 token vs 1024 token 太小丢失上下文，太大引入噪声
向量索引算法 HNSW vs IVF-PQ 精度与速度的权衡，HNSW 更准但内存消耗大
检索数量 K Top-3 vs Top-10 K 越大召回越全，但 Token 成本和噪声也越高
嵌入模型 通用模型 vs 领域微调 医疗、法律等专业领域用领域微调效果显著提升
Chunk 大小是 RAG 中最常见的坑：很多初学者把整篇文档直接塞进向量数据库，结果检索到的片段过于宽泛，包含大量无关信息。记住：chunk 的目标是让每个片段刚好包含一个完整的信息单元——既不要碎片化，也不要当百科全书。

RAG 与上下文窗口的关系

RAG 和 LLM 的上下文窗口（Context Window）是互补的关系，不是竞争关系。

RAG 相当于"海选"——从海量文档中筛选出最相关的片段。上下文窗口相当于"精读"——把筛选后的精华塞进模型的可读范围内。即使上下文窗口扩展到 1M+ token（如 Gemini 1.5 Pro），RAG 仍然不可替代，因为：① 减少噪音干扰（太多无关信息会导致模型忽略重点）② 成本控制（100 万 token 的推理成本远高于检索 1000 token）③ 实时性（数据库秒级更新 vs 模型重训数周）。

为什么 RAG 成为企业 AI 的事实标准

在 ChatGPT 上线之初，企业面临一个核心问题：如何让 AI 访问私有数据？RAG 给出了最优雅的解决方案——不改动模型本身，只在模型前面加一层检索系统。

Salesforce 的 Einstein GPT 将 RAG 作为核心架构，连接 CRM 中的客户数据。LangChain 和 LlamaIndex 等开源框架将 RAG 封装为标准化工具链。研究表明，在专业问答场景中，RAG 可以将事实性准确率从基线的 50-60% 提升到 75-85%。

在个人层面，RAG 被用来构建"和你的知识库对话"——把读过的书籍摘要、课堂笔记、研究论文存入向量数据库，再用 RAG 实现个性化搜索。这种应用的边际成本极低，存储开销可以忽略不计。

跨学科连接

RAG 的设计原理与多个学科的核心思想相通。认知科学中的双系统理论可以类比 RAG 架构：系统 1（快速、直觉）对应 LLM 的生成能力，系统 2（慢速、推理）对应检索过程——人类在做重要决策时也会先"检索"记忆再作答。信息论中的信源编码理论指出，信息传递需要在压缩率与保真度之间权衡——RAG 的 chunk 大小选择本质上就是在做这种权衡。图书馆学中的信息检索体系直接启发了 RAG 的索引策略，从杜威十进分类法到现代语义搜索，核心挑战完全同构。
